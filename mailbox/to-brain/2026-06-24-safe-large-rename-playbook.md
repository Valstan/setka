# Находка: безопасный сквозной rename термина через audit-workflow + placeholder-protect

## Контекст

Владелец попросил убрать слово «дайджест»/`digest` из **всей** системы (people не нравится,
и по сути неверно — у нас полное копирование новостей, не выжимки). Замена: публикации →
«сводка», код → `bulletin`. Масштаб: **~1700 вхождений в ~130 файлах** (БД-таблицы, env-ключи,
имена Celery-задач, метрики Prometheus, API-роуты, JS фронтенда, тесты, доки, проза).

Слепой `sed digest→bulletin` сломал бы прод: переименовал бы stdlib (`hexdigest`), имена
метрик (история Grafana), Redis-ключи (непрерывность cooldown), **персистентные JSON-ключи в
БД** (данные осиротеют), имена Celery-задач (beat-расписание). Сделали за 5 гейтнутых PR + 2
миграции, **0 регрессий** (1432 теста зелёные на каждом шаге).

## Переносимый паттерн (для любого проекта с большим rename)

**1. Тиражируй по риску, не по алфавиту.** 5 слоёв отдельными PR:
- видимый текст (читатель) → видимый текст (оператор/UI) → идентификаторы кода →
  прод-контракты (БД/env/задачи, под подтверждением) → long-tail (переменные/комментарии).
- Каждый PR — свой гейт (lint+тесты+CI), деплой батчем после видимых фаз.

**2. Перед слепым rename — audit на персистентность (это и есть ключевой трюк).**
Прогнали **workflow из 15 субагентов** (классификаторы по группам файлов + adversarial-
верификаторы + completeness-critic): для каждого `digest`-токена — «персистентный/контрактный
или внутренний?». Каждый токен трассировался до хранилища (DB-колонка, JSON-ключ в Column(JSON),
Redis/cache-ключ, имя метрики, имя Celery-задачи, stdlib). Нашёл ровно **один** скрытый
персистентный ключ (`regions.config->>'digest_mode'`, 2 ряда), который иначе молча осиротил бы
маршрутизацию. Без adversarial-прохода его бы пропустили.

**3. Placeholder-protected слепой рейн.** Защищённые токены (stdlib + метрики + Redis-ключи +
имя файла миграции) → заменяем на NUL-плейсхолдеры → слепой `DIGEST/Digest/digest →
BULLETIN/Bulletin/bulletin` по case-вариантам → восстанавливаем плейсхолдеры. Грамматика
русского (м.→ж. род: «каскадный дайджест»→«каскадная сводка») требует **фразовых** правил, не
пословных — иначе «каскадный сводка»/«Сводка пуст».

**4. Внутренние «контракты» — не контракты, если все потребители в репо.** Adversarial-
верификатор отсеял ложные тревоги: dict-ключи/Pydantic-поля, чьи потребители (тесты, скрипты,
шаблоны) переименовываются тем же проходом — безопасны. Реально опасны только пересечения
наружу: stored data, внешний HTTP-клиент, stdlib, persisted метрика/ключ.

**5. Verify-after: residual-grep.** После — grep остатка термина: должен быть **только**
защищённый набор. «Полный ноль» недостижим (stdlib `hexdigest` остаётся) — и это нормально,
документируем why.

## Кандидаты на переиспользование

Любой rename термина/идентификатора в GONBA / MatricaRMZ / SabantuyMalmyzh (Drizzle/Payload —
там вместо «персистентный JSON-ключ» будут колонки/локали, но логика audit-before-blind та же).
Родственно G23/G74 (drizzle rename-замки) — там инструмент **спрашивает** про rename; здесь мы
**сами** доказываем безопасность через audit-workflow до мутации.
