|---|---|---|
| Правки/ветки/коммиты | авто | — |
| Push feature-ветки, `gh pr create`, `gh pr merge --squash` | авто | `pre-commit` + `pytest` + CI зелёные |

## Две находки по инструменту (переносимо — кандидаты в GOTCHAS)

Раскатывая #027 на Claude Code, упёрся в два ограничения системы разрешений, которые меняют дизайн settings.json в любом проекте:

1. **Правила сопоставляются ПО ИМЕНИ ИНСТРУМЕНТА.** `Bash(pytest:*)` НЕ покрывает ту же команду, запущенную через инструмент `PowerShell` — нужен отдельный `PowerShell(pytest:*)`. На Windows-машинах (а у владельца обе локальные — Windows/PowerShell) без `PowerShell(...)`-дублей автономия молча не срабатывает: команды всё равно спрашивают. В setka в `allow`/`deny` теперь оба набора.
2. **Сопоставление только ПО ПРЕФИКСУ** (`:*` валиден лишь в конце; mid-string `*` ненадёжен). Следствие для #025: `deny`-правило **не может** отличить read-only `ssh setka "..."` от destructive `ssh setka "...psql DROP..."` — дискриминатор внутри кавычек, недосягаем для префикс-матчинга. Поэтому черту #025 на setka держит **поведение** (обязательный `AskUserQuestion` перед необратимой прод-операцией, + гейты `/sql` DML и `/reliz`), а не deny-правило. Это, видимо, общий вывод для всех проектов: «destructive-прод гейт #025 не выражается машинно через permissions для SSH-туннелированных команд → остаётся поведенческим».

Если полезно — оба пункта тянут в GOTCHAS (G-«permission rules per-tool-name» и G-«prefix-match не различает destructive внутри ssh-кавычек»). На твоё усмотрение.

## Статус

`allow`. Блокеров нет. Деплой-ярус на setka остаётся полу-гейтнутым осознанно: типовой релиз несёт миграцию на прод-данные (#025), поэтому `/reliz` ведёт миграции под подтверждением; полностью авто — только не-схемные правки.
