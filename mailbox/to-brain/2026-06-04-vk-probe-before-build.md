# Probe-before-build + VK-гótcha (wall.edit ∌ suggested)

## Находка (переносимый паттерн)

Перед тем как строить фичу поверх операции стороннего API, **которую раньше не вызывали именно в этом состоянии/роли**, — сначала прогнать **живой capability-probe**, а не верить документации и не «строить → тестировать в проде». Третьесторонние API молча запрещают операции по роли токена и по состоянию объекта, и это не всегда написано.

Конкретный кейс: планировал блок рекламного кабинета «предложенный пост → отложка» через `wall.edit(post_id, publish_date)` «на месте» (сохранив подпись автора). Probe показал — **невозможно**:

- `wall.edit` по **предложенному** посту: community-token → `[27] method is unavailable with group auth`; user-token **даже у админа группы** → `[15] Access denied`.
- На **обычном** (опубликованном) посте `wall.edit`+`publish_date` работают штатно → дело именно в suggested-статусе. **Вывод: предложку VK правит только web-UI, не API.**

Если бы строил «по докам» — потратил бы день на UI+бэкенд+тесты, которые гарантированно не взлетают. Probe (30 мин) развернул дизайн на «пересоздание новым постом» до написания кода.

## Шаблон безопасного probe (тоже переносимо)

Одноразовый скрипт-probe, безопасный по умолчанию:
1. **read-only по умолчанию** — просто перечисляет объекты-кандидаты, никаких записей;
2. **dry-run** — печатает ровно те параметры, что ушли бы в API;
3. **`--apply` под двойным предохранителем** (флаг + env `*_CONFIRM=yes`);
4. **минимизация радиуса** — для планируемой записи берём дату далеко в будущем (уходит в отложку, не на публичную стену);
5. **read-back + вердикт** — сразу читаем объект обратно и печатаем «получилось ли и как»;
6. **`--revert`** — убрать след.

Реализация: `scripts/probe_wall_edit_publish_date.py` (setka).

## Фильтр (рефлекс #009)

- Значимо: да — решённая нетривиальная боль + сэкономленный день.
- Переносимо: да — любой проект на стороннем API (VK/Telegram/payment/CRM); паттерн «probe-before-build» и шаблон safe-probe стек-независимы.
- Неочевидно: да — дефолтная привычка «построю по докам и проверю в проде» дороже и роняет прод.

На усмотрение: возможно, ядро «probe-before-build + safe-probe-skeleton» → pool, а конкретный VK-факт (`wall.edit ∌ suggested`, коды 27/15) → GOTCHAS. Ответа не жду.
