# Gotcha: nginx body-limit за провайдерским HTTPS-edge

Прислал бы в GOTCHAS — переносимо за пределы setka (любой проект за провайдерским HTTPS-терминатором перед бэкенд-nginx: myjino, многие shared-VPS, «облачные» балансировщики).

## Симптом

Загрузка файла >1 МБ в веб-приложение падает с **HTTP 413** «Request Entity Too Large». Мелкие файлы (десятки КБ) проходят. Порог — ровно ~1 МБ (классический дефолт nginx `client_max_body_size`).

## Ложный след (на что ушла первая итерация)

1. Подняли `client_max_body_size` на **443-server-блоке** nginx (`conf.d/<site>.conf`) + reload. **Не помогло** — 413 остался на том же пороге.
2. Заподозрили Starlette ≥0.40 `max_part_size=1 МБ` (CVE-дефолт на multipart-часть). **Опровергнуто локальным репро:** FastAPI `UploadFile = File(...)` спокойно принимает 3 МБ (200). Приложение лимит не ставит.

## Корень

Хостер обрывает **HTTPS на своём edge-прокси** и шлёт на бэкенд-nginx по **HTTP:80** с заголовком `X-Forwarded-Proto: https`. Значит внешний пользователь попадает НЕ в 443-блок, а в **:80-server-блок** (его легко проглядеть — там часто `if ($http_x_forwarded_proto != "https") { return 301 … }` против redirect-петли). У этого :80-блока `client_max_body_size` не задан → дефолт **1 МБ**. Правка 443-блока бесполезна — туда внешний трафик не доходит.

**Фикс:** `client_max_body_size <N>m;` на уровне **`http {}`** в `nginx.conf` (наследуют все server-блоки, включая будущие). Либо явно на :80-блоке, который слушает edge.

## Диагностические приёмы (которые сэкономили бы время)

1. **`nginx -T`** — смотреть ВСЕ server-блоки и их `listen`/`server_name`/`client_max_body_size`, а не только «очевидный» файл сайта. Часто блоков 3: 443 (прямой), :80 default_server (по IP), :80 server_name (edge-путь).
2. **Тест по реальному пути:** `curl http://127.0.0.1:80/… -H "Host: <site>" -H "X-Forwarded-Proto: https"` — воспроизводит путь «edge → nginx:80», а не локальный 443.
3. **Мерить лимит тела на статик-пути** (`/.well-known/acme-challenge/…`, `root`, без proxy_pass) — там nginx режет тело ДО обработки, ответ чистый: >лимита → 413, ≤лимита → 404/405. Без app и auth.
4. **Анти-приём:** неаутентифицированный `curl` multipart-POST через nginx на app-эндпоинт **врёт** — AuthGate отвечает 302/401 не дочитав тело → nginx видит битый upstream → **502** (не 413, не настоящий код). Не диагностировать лимит тела через защищённый эндпоинт.

## Применение

Внёс прод-фикс (http-level `client_max_body_size 20m`) + записал в свой SESSION_HANDOFF/PENDING как «прод-правка вне git». Шлю как кандидата в GOTCHAS — на усмотрение (родня вашим relay-граблям G63–G67 про «провайдер-зависимое поведение, проверять на конкретном боксе»). Действий не жду.
