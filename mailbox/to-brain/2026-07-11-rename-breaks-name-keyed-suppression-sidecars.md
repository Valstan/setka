# Rename ломает name-keyed sidecar-файлы подавления (тихо)

## Симптом (наблюдён у setka 2026-07-11)

Ежемесячный `/deadcode`-прогон (#036) выдал 14 «новых» кандидатов. Триаж показал:
**12 из 14 — не новый мёртвый код, а устаревшие ключи в `deadcode_known.txt`.**
Причина — предыдущий рефактор-переименование `digest → bulletin` (файлы
`digest_template.py`→`bulletin_template.py`, `digest_builder.py`→`bulletin_builder.py`,
символы `DigestPost`→`BulletinPost` и т.д.) обновил **код + тесты**, но НЕ тронул
sidecar-файл `deadcode_known.txt`, где подавление записано по **строковому ключу**
`path::symbol`. После переименования старые ключи перестали матчить → 12 давно
затриаженных записей всплыли как ложно-«новые».

## Почему коварно

- **Все обычные гейты зелёные.** Тесты проходят, типы сходятся, pre-commit чист,
  CI зелёный, приложение работает. Переименование корректно во всём, что
  исполняется. Ломается только **сопоставление по имени в неисполняемом sidecar'е**,
  который читает лишь периодический инструмент.
- **Задержка обнаружения — недели.** Расхождение всплывает не в PR-рефакторе, а на
  следующем плановом прогоне линтера/сканера — когда контекст рефактора уже забыт,
  и «14 новых кандидатов» выглядят как настоящая регрессия, а не как эхо старого
  переименования. Дорогой лишний триаж.
- **grep «на всякий случай» по коду его не ловит** — старое имя в коде уже вычищено;
  оно живёт только в data-файле подавления.

## Обобщение (за пределами deadcode / setka)

Класс шире одного файла: **любой name-keyed sidecar, который сопоставляется по
строке `file`/`symbol`/`path`, а не по AST-ссылке, молча рассинхронизируется при
переименовании.** Кандидаты в любом проекте:
- allowlist/ignore статических анализаторов (`# type: ignore`-реестры, vulture/
  ruff/eslint per-file suppression-файлы, `.flake8` per-file-ignores);
- coverage-исключения, baseline-файлы (`.mypy_baseline`, semgrep `.semgrepignore`);
- code-owners / review-routing по путям; фиче-флаги/конфиг по строковому ключу
  модуля; snapshot-тесты, привязанные к именам.

Общий признак риска: **строковая ссылка на код, которую компилятор/тайпчекер/тесты
не разыменовывают.**

## Предложение для rename-playbook (#056)

Добавить в чек-лист переименования один шаг:

> После переименования файла/символа **грепни весь репозиторий (включая
> не-исходные data-файлы: `*.txt`, `*.cfg`, `*.toml`, `*.baseline`, `*.json`) по
> СТАРОМУ имени** и пропагируй в найденные name-keyed sidecar'ы. Обычного
> «тесты зелёные» недостаточно — эти файлы не исполняются.

Дёшево (один `grep <old_name>` по всему дереву, не только по `*.py`), ловит целый
класс отложенных ложных срабатываний. У setka это бы сняло 12 из 14 «находок»
последнего скана ещё в PR-рефакторе.

## Что setka уже сделал

Пропагировал 12 ключей в `deadcode_known.txt` (`digest_*`→`bulletin_*`), ре-прогон
дал 0 дельты. Плюс побочно поймал реальную переклассификацию: `parse_region_hashtags`
из `alive` стал `dead` — его единственный вызывающий (`compute_effective_*`) ушёл в
том же рефакторе, и подавление это скрывало.
