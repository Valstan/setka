## 3. Анти-бан дисциплина — энфорс

Большинство требований шлюз уже наследовал от токен-переустройства #336/#337
(тот самый #072, применённый к общему пулу):

- ✅ **карусель `last_used ASC`** — `TokenPolicy.pick(READ)` сортирует по
  последнему использованию, нагрузка равномерна по VALSTAN+VITA+МАМА;
- ✅ **самолечение + авто-cooldown + alert** — VK error 5/17/29 → cooldown
  токена + Telegram-alert, шлюз перебирает следующего кандидата;
- ✅ **per-consumer квота** — 30/мин + 5000/сутки на ключ → 429 + Retry-After
  (было с v1), отказ логируется;
- ✅ **НОВОЕ: per-credential (агрегатный) бюджет** — `GATEWAY_GLOBAL_QUOTA_PER_MIN`
  (дефолт 120/мин суммарно по всем потребителям, ниже лимитов VK с запасом);
  превышение → 429 потребителю ДО расхода токена, НЕ проглатывание. Порядок:
  сначала per-consumer (долбящий получает свой 429, не выедая общий бюджет),
  потом агрегат.

**Отложено с обоснованием:**
- *Upfront probe-before-use на каждый запрос шлюза* — не делаю: probe удваивает
  VK-вызовы (сам probe тратит бюджет токена — анти-бан-антипаттерн для
  высокочастотного пути). Ленивое самолечение (перебор кандидатов по
  cooldown-кодам с report_error) даёт ту же защиту без лишних вызовов; upfront
  probe остаётся в низкочастотных путях (`pick_healthy_read_token` парсинга).
- *Каскад эшелонов на write-пути* — к шлюзу неприменим: шлюз read-only (v2
  write — отдельная owner-gated нитка, scope не решён).
