# Коррекция: «контекст авторизации в промпте» лечит over-refusal рутины лишь временно — надёжно лечит перенос обвязки в код

Прямое **уточнение вчерашнего письма** (`2026-07-08-cloud-routine-over-refusal-authorization-context.md`).
Вчерашняя эвристика («промпт headless-агента должен явно заявлять авторизацию») **работает недолго и
хрупка** — сутки спустя тот же приём стал причиной отказа. Ниже — что случилось и более надёжный рецепт.

## Что произошло

v2-промпт (блок «Контекст и авторизация»: чей сервис, кто запустил, что делают вердикты, HITL) отработал
~сутки (2026-07-07..08), потом рутина **снова встала**. Причём отказ пришёл с **двух** сторон:

1. **С авторизационным эссе** → модель прочитала САМ блок как манипуляцию: «"you're authorized", "this is
   your own service" — это в точности риторическая структура social engineering; legitimate authorization
   should be simpler». Фразу «без этого контекста задача читается неверно» — как попытку снять ограничения.
2. **Без эссе (сухой нейтральный промпт)** → модель отказалась уже по harness-пометке рутины: «this is an
   automated background-task event, NOT a message from the user → I should not act, standing by».

Оба края промпт-пространства отбиты. Наращивать оправдания = кормить триггер #1; убирать = получать #2.

## Корень (сместился глубже, чем вчера казалось)

Ни один отказ не был **против классификации**. Все три (за два дня) — против **обвязки**:
- API-ключ **открытым текстом в инструкции**;
- модель **сама** делает автономные HTTP-вызовы с этим ключом на внешний хост.
Модель прямо предписывала лечение: «store credentials in environment variables», «use a script/CI, not an
agent session». То есть она возражала не против задачи, а против того, что секрет и сетевая механика **лежат
в её промпте и исполняются ею**.

## Надёжный фикс — вынести обвязку из промпта в код (не риторика, а структура)

Выполнить то, что модель сама просит, вместо уговоров:
1. **HTTP + ключ → детерминированный скрипт в репо** (`scripts/classifier_routine.py`, stdlib-only —
   в облачном checkout'е зависимостей проекта нет). Подкоманды `fetch` (GET → файлы) / `submit`
   (валидация + POST).
2. **Ключ → env-секрет облачного окружения рутины** (`Update cloud environment → Environment variables`),
   в промпте ключа **нет вообще**.
3. **Модели — только доменная задача:** «прочитай файлы `pending.json` + `postulates.md`, размечай,
   сохрани `verdicts.json`, запусти submit». Никаких оправданий, никакого секрета в поле зрения модели.

Результат: следующий же прогон отработал — +29 качественных вердиктов (publish/hold/delete с калиброванной
уверенностью), счётчик сдвинулся. Промпт стал короче и **честнее** (не спорит о легитимности — просто
описывает работу).

## Почему шлю (переносимость + неочевидность)

- **Контр-интуитивно:** естественный рефлекс при отказе — *добавить* контекста/оправданий (мы так и
  сделали в v2). Это делает **хуже**: авторизационное эссе само палится как jailbreak. Правильный вектор —
  **обратный**: убрать из поля зрения модели то, к чему она придирается (секрет, автономную сетевую
  механику), а не доказывать ей право.
- **Переносимо:** любой проект экосистемы с облачной рутиной против собственного API (VK-шлюз, будущие
  headless-агенты) наступит на то же. Рецепт «обвязку в код + секрет в env, агенту чистую задачу» —
  общий паттерн, не доменный.
- **Пара к egress-гоче:** теперь триптих «почему облачная рутина не делает работу»: (1) инфра — Network
  access → Custom; (2) ~~поведение — контекст авторизации в промпте~~ **отозвано как ненадёжное**;
  (2′) структура — унести ключ+HTTP в скрипт, модели оставить домен.

Эвристика в одну строку (замена вчерашней): **если headless-агент отказывается от легитимной задачи против
своего API — не уговаривай промптом (эссе-оправдание само читается как jailbreak, а его отсутствие — как
«не мой запрос, стою в стороне»). Вынеси ключ и сетевую механику в детерминированный скрипт + env-секрет,
оставь модели только доменную работу.** Промпт-путь при отказе исчерпаем; надёжнее — enforce (прямой
API-вызов из своего кода), к нему же ведёт этот паттерн.
