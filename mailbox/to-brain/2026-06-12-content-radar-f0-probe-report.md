# Probe-отчёт Ф0 контент-радара (#020, до постройки)

Probe выполнен 2026-06-12 (скрипт — `scripts/probe_tme_s_parsing.py`, повторяемый,
stdlib-only: гоняется и локально, и на проде через `ssh setka "python3 -" < …`).
Четыре блока фактов, два требуют решения владельца.

## 1. Парсинг `t.me/s/<канал>` — механика РАБОТАЕТ, но НЕ С НАШЕГО VPS

**С обычной машины (Windows, домашний провайдер)** — всё зелёное:

| Проверка | Результат |
|---|---|
| Доступность | 200 на всех каналах (свои `malmyzh_info`/`gonba_life` + жирный `tass_agency`) |
| Полнота страницы | 6–20 сообщений/страница; у каждого `data-post` id, `<time datetime>`, текст-блок; фото/видео различимы по CSS-классам |
| Пагинация | `?before=<min_id>` стабильно отдаёт строго более старые id, 3 страницы вглубь без сбоев |
| Несуществующий/закрытый канал | детектится по redirect `t.me/s/x` → `t.me/x` (не по 404 — статус всё равно 200) |
| Burst rate-limit | 8 запросов подряд без пауз — все 200, ~1 c каждый, троттлинга не видно |

**С прод-VPS (myjino, RU) — Telegram заблокирован на уровне сети**, причём не DNS, а IP
(TCP connect к 149.154.167.99:443 — timeout; общий outbound жив: VK 200, Google 200):

| Хост | С VPS |
|---|---|
| `t.me` / `telegram.me` / `telegram.dog` / `web.telegram.org` | ❌ timeout (errno 101 / connect timeout) |
| `cdn4.cdn-telegram.org` (медиа!) | ❌ timeout |
| `api.telegram.org` (Bot API) | ✅ работает (им и шлём зеркала/алёрты сейчас) |

Bot API не читает чужие каналы (бот должен быть админом) — это не обход.

**Следствие: TG-source-адаптер не может поллить t.me/s/ напрямую с tiny.** Опции
(решение за владельцем/тобой, в порядке моей рекомендации):

1. **Egress-relay для t.me** — дешёвый внешний фетчер: serverless (Cloudflare Workers
   free-tier: ~100k req/день — на порядки больше нужного) или копеечный зарубежный
   прокси в env (`TG_PREVIEW_PROXY`, #008). Relay обязан проксировать и **медиа-байты**
   (CDN тоже заблокирован — иначе save-в-архив картинок TG не будет работать).
   Адаптер остаётся pluggable, ядро не трогаем; relay — деталь конфига адаптера.
2. **Сдвинуть TG-источники в Ф1**, Ф0 выпустить на VK + RSS (оба работают с VPS
   без обвязки). Минус: TG-каналы — половина мотивации радара.
3. Поллер TG на другой машине флота — у всех боксов тот же RU-хостинг, не вариант
   (отметаю).

## 2. Web-push с tiny — зелёный

- Push-endpoints достижимы с VPS: `fcm.googleapis.com` ✅, `web.push.apple.com` ✅,
  `updates.push.services.mozilla.com` ✅ (TCP+TLS живые, коды 404/405/406 на пустой
  GET — норма).
- `pywebpush`/`cryptography` в prod-venv нет — обычный `pip install` в Ф0, не блокер.
- Вердикт: **web-push в Ф0 возможен**, лента+push (не «только лента»).

## 3. Домен для PWA — УЖЕ ЕСТЬ, вопрос закрыт фактом

Письмо предполагало «setka headless, домена нет» — реальность лучше и хуже сразу:

- ✅ nginx уже слушает 443 на техдомене `3931b3fe50ab.vps.myjino.ru` c валидным
  wildcard-LE-сертификатом jino `*.vps.myjino.ru` (до 2026-07-29, авто-renew их
  стороной — ровно G20). Внешний HTTPS → 200. **Для Ф0 PWA техдомена достаточно**,
  свой домен — к публичному запуску (как и писал ты).
- ⚠️ **Но это значит, что операторский SETKA UI прямо сейчас доступен из интернета
  без какой-либо авторизации**: Dashboard, `/api/health/full`, страница `/tokens`
  (управление VK-токенами) — всё отдаёт 200 любому, кто знает/просканит hex-домен.
  Auth'а в setka нет вообще (исторически считалась headless за SSH).
  **Это поднимает приоритет «auth + изоляция» из Ф0-требования в немедленный
  security-фикс**: до постройки полноценного auth — хотя бы nginx basic-auth /
  IP-allowlist на 443 (предложил владельцу в этой же сессии).

## 4. Sizing-факты для архива (Ф0-схема с учётом байтов)

- Диск tiny: 9.9 GB всего, **4.3 GB свободно** (55% занято) — стартовая квота
  архива должна быть консервативной (например, дефолт 200–500 MB/юзер,
  предупреждение с порога — механика квот всё равно Ф1+).
- RAM 1.5 GB (из probe 06-07) — без изменений, embeddings-фичи не закладываем.
