# «Сменный мотор» LLM-классификации + грабля «пустого источника»

Достроил HITL-классификатор (ADR-0003). По ходу вылезли две переносимые вещи — делюсь по рефлексу #009.

## 1. Паттерн: сменный мотор классификации за стабильной основой

Владелец не захотел прямо сейчас заводить `ANTHROPIC_API_KEY` (спенд/новый billing), но хотел начать
собирать shadow-данные. Решение (вариант B): **вся основа не зависит от того, чем классифицируем**, а
«мотор» — сменный.

- **Стабильная основа** (строим один раз): shadow-таблицы вердиктов + лога коррекций; **HTTP-ingest**
  (`GET /pending` батч → `POST /verdicts`, X-API-Key); операторская лента с agree-rate по типам;
  файл-корректировщик; shadow-гейт.
- **Мотор сегодня:** scheduled cloud agent (Claude Code routine на claude.ai/code) по крону ходит в ingest,
  классифицирует, возвращает вердикты. Крутится в облаке → не зависит от включённого компа, ноль
  permission-промптов на боксе (лечит ровно те боли, из-за которых владелец свернул локальные рутины).
- **Мотор завтра:** Celery-таск + Claude API пишет в **те же** таблицы через тот же сервисный слой;
  ingest глушится kill-switch'ем. Основа/лента/метрика/файл правил не меняются.

**Почему переносимо:** любой проект экосистемы, которому нужна LLM-классификация/разметка, но не готов
сразу на метрику API-биллинга, может стартовать с облачной рутины за HTTP-ingest, а потом бесшовно
переключиться на API — без переписывания UI/хранилища/метрики. «Отложи выбор мотора на активацию»
(родня choice-independent push/pull из R17). Кандидаты: trener/GONBA/малмыж-сайты, если захотят
LLM-разметку контента.

**Тонкость (не на поверхности):** ingest должен быть **эхо-само­достаточным** — рутина возвращает не
только вердикт, но и снапшот исходника (текст/url/ключ), потому что источник может быть транзиентным
(см. п.2). Тогда переключение мотора и переигровка не зависят от того, жив ли ещё исходник.

## 2. Грабля: проверять, что источник РЕАЛЬНО наполнен, до постройки на ORM-таблице

Построил `/pending` поверх модели `Post` (она есть в схеме, монитор её пишет). При деплое `/pending`
вернул 0: таблица `posts` **пуста** — активный конвейер SARAFAN давно не пишет пер-пост Post-строки
(`monitor.py` не в celery beat), а копит кандидатов внутри свод­ок (`bulletin_curation_runs.candidates`).
Перенаправил на реальный источник, сменил ключ идентичности на структурный фингерпринт (`lip`).

**Урок (переносимо для любого агрегатора/пайплайна):** наличие ORM-модели ≠ живой источник. Перед
постройкой на таблице — `SELECT count(*)` + свежесть на проде, а не доверие схеме. Дёшево, экономит
переделку (у меня — миграция 054 + переписанный слой). Кандидат в GOTCHAS, если решишь.

Оба — `suggest`. Promote в pool имеет смысл после первого реально-переключённого 2-го потребителя
(measure-before-promote), пока REFERENCE-grade.
