# Две переносимые граблы (сессия 2026-06-14)

Обе всплыли при постройке/QA фич, обе применимы за пределами setka. По фильтру
рефлекса #009 (значимо ∧ переносимо ∧ неочевидно) — проходят.

## Грабля 1 — env-значения с shell-метасимволами: systemd OK, bash `source` ломает

**Симптом.** Добавил в `/etc/setka/setka.env` строку с пробелами/`{}`/`!`/«»
(текст шаблона: `AD_AUTO_GREETING_TEXT=Здравствуйте, {author_name}! …`). Сервисы
(systemd `EnvironmentFile=`) работали идеально — значение читалось целиком. Но
любой скрипт по паттерну `sudo bash -c 'set -a; source /etc/setka/setka.env; …'`
(у нас так гоняется `/curate` и probe-скрипты) спотыкался:
`{author_name}!: command not found`, а сама переменная обрезалась до первого
пробела.

**Корень.** `systemd EnvironmentFile` парсит `KEY=VALUE` **буквально** (вся
строка после `=` — значение). `bash source` **интерпретирует** строку как код:
`KEY=слово1` (присваивание до пробела), дальше `слово2 слово3…` исполняется как
команда. Два разных парсера одного файла → значение валидно для одного и
сломано для другого. **Тихо**, потому что прод-сервисы (systemd) не падают —
ломаются только вспомогательные sudo+source-скрипты, и то «мягко».

**Лечение.** Значения с пробелами/`!`/`{}`/«»/`$` — в **двойные кавычки**:
`KEY="…"`. systemd кавычки срезает (значение то же), bash читает корректно.
Нюанс: `KEY=*` без кавычек безопасно (в присваивании bash не делает globbing).

**Переносимо** на любой проект, где (а) прод-секреты в env-файле, (б) systemd
`EnvironmentFile`, (в) есть скрипты, читающие тот же файл через `source`. Правило
в одну строку: **«env-значение с любым не-словарным символом — сразу в кавычки»**.
Кандидат в GOTCHAS.

## Грабля 2 — одно-кнопочные прод-мутации: check-then-act не атомарен → дубль на двойной клик

**Контекст.** Делал/ревьюил эндпоинты «одной кнопкой» для рекламы:
моментальный `wall.post` в живую стену и сетевую рассылку по 16 пабликам.
Классический паттерн: `row = get(); if row.status == 'done': return; …act…;
row.status='done'; commit`.

**Грабля.** На async-сервере (FastAPI/uvicorn) два конкурентных запроса (двойной
клик, ретрай) **оба читают `status='new'` до коммита первого** — интерливинг
происходит на `await` внешнего вызова (сам `wall.post`/`messages.send`). Оба
постят → **дубль в реальном паблике / двойная ЛС**. Последовательный повтор
guard ловит (второй видит `done`), а конкурентный — нет. Коварство: «коммитим
статус сразу после поста» выглядит как защита, но защищает только от *повторного
вызова*, не от *одновременного*.

**Лечение (два слоя):**
1. **Атомарный клейм перед действием.** Либо `SELECT … FOR UPDATE` (второй
   запрос блокируется на строке до коммита первого → видит `done` → выходит),
   либо `UPDATE … WHERE id=? AND status!='done'` с проверкой `rowcount`, либо
   `INSERT … ON CONFLICT DO NOTHING` для «одно действие на (сущность, прогон)».
2. **Reclaim зависших claim'ов.** Если клеймишь *до* действия (claim='pending'
   → act → mark done), крах между claim и mark **навечно вешает** сущность
   (claim есть, done нет → больше никто не возьмёт). Нужен sweep: claim старше
   grace (≫ длительности действия) → пометить terminal-error (НЕ
   переисполнять — статус внешнего эффекта неизвестен → анти-дубль), чтобы
   процесс мог продолжиться. Иначе рестарт/деплой mid-action = тихий вечный
   stuck + ложные watchdog-алёрты.

**Переносимо** на любой проект с операторскими «одной кнопкой» действиями,
мутирующими внешнее состояние (постинг, отправка, оплата, заказ). Правило:
**«любая кнопка, делающая необратимое во внешний мир, обязана клеймить атомарно,
а claim-before-act — ещё и реклеймить зависшие claim'ы»**. Кандидат в GOTCHAS /
REFERENCE (паттерн идемпотентной одно-кнопочной прод-мутации).

переносимо — пульни в pool/GOTCHAS; если нет — просто оставь себе на радаре.
