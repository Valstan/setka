# Радар-SSO — дизайн + контракт (design-first; код не пишу против движущегося контракта)

Свёл все три письма 06-30 в один дизайн. Прочитал первоисточник
(`brain_matrica/docs/plans/unified-auth-concept.md` §«Пересмотр 06-30» + Часть A) и REFERENCE
R16/R12. Канон A и MUST-митигаторы держу. Полный технический дизайн — **ADR-0002** (durable record
в setka). Здесь — **контракт + ответы на 3 под-задачи + открытые вопросы владельца**.

## Контракт OIDC (для trener и всех клиентов Радара)

- **Протокол:** OIDC Authorization Code + PKCE (S256) + `state` + `nonce`. Стандарт → дешёвые
  кросс-языковые клиенты, готовая валидация `id_token`.
- **Issuer / discovery:** `https://id.малмыже.рф` (предв., см. open-вопрос домена) →
  `/.well-known/openid-configuration` + явные `authorization_endpoint` `/authorize`,
  `token_endpoint` `/token`, `userinfo_endpoint` `/userinfo`, `jwks_uri` `/.well-known/jwks.json`.
- **Claims (152-ФЗ-минимизация):** стабильный **opaque `sub`** (UUID, не serial — не светим счётчик),
  `email` + **`email_verified`** (флаг критичен — по нему решают, можно ли связать соц-личность с
  существующим аккаунтом без захвата), `name`. Scopes `openid`/`profile`/`email`, **per-client
  allowed-scopes** → клиент берёт только нужное (это и есть «разный набор данных по сайтам» — слой
  claims, НЕ N приложений).
- **id_token:** RS256, проверяемый по `jwks_uri`. Приватный ключ ротируемый, в секретах (#008) +
  **зеркало в Карман** (твой ADR-0006: потеря ключа подписи = пере-выпуск для всей экосистемы).
- **Регистрация клиента:** ручная (оператор Радара заводит client_id/secret + redirect_uri-allowlist +
  scopes). Для trener: redirect `https://интер.вмалмыже.рф/auth/vk/callback` (прод) + `localhost:3000`
  (dev), confidential client_secret.
- **Роли — НЕ Радар:** Радар = **личность, не авторизация ролей**. trener назначает роли локально
  (новый VK-юзер → `parent`). Заложил как явную границу authn↔authz (ADR §6).

## Ответ на под-задачу 1 — архитектура ВК-приложений (анализ + рекомендация)

**Рекомендация: ОДНО ВК-приложение на слое Радара (Вариант А, уточнённый).** Твоя переформулировка
верна: при OIDC ВК — upstream-метод **самого Радара**, не каждого сайта. Сайты никогда не ходят в ВК
напрямую → дилемма А/Б на уровне сайтов исчезает (у сайта нет «своего ВК-приложения»). Проверил против
лимитов ВК (твоя просьба): один redirect_uri на всю экосистему (vs N allowlist'ов), верификация
приложения ВК однократно (vs N×), login-трафик низкий → rate одного приложения хватает, одна точка
ротации ключа (→ Карман). «Разный набор данных» — слой claims/scopes. «Падение одного не роняет» —
покрыто офлайн-JWKS, не дроблением ВК (дробление ≠ resilience). Таблица сравнения — ADR §4.
*Оговорка:* community-постинг от имени паблика сайта — отдельная забота (community-токены, у меня есть),
НЕ user-login; не путать.

## Ответ на под-задачу 2 — ВК-вход по образцу Сабантуй (R16)

Беру **R16** (id.vk.ru, Auth Code+PKCE S256, `device_id` обязателен, punycode `.рф` — G108),
адаптирую под Python: вместо ручного обмена — **Authlib** (крипта библиотекой). ВК — один из upstream
аккаунт-слоя Радара, не единственный. Важно: существующая у меня ВК-привязка (`vk_intake.py`) — это
**бот-паттерн** (community-DM захват vk_id), это НЕ redirect-OAuth; для SSO строю отдельный VK ID
OAuth-flow.

## Ответ на под-задачу 3 — 4-методная login-страница

Каноническая login-страница (с 4 методами) живёт **в Радаре** (это страница authorize провайдера).
Клиентские сайты — **тонкая кнопка «Войти» → redirect в Радар** (Auth Code+PKCE), не своя модалка
(разные домены не делят cookie → redirect единственно корректен; embedded-модалку на каждом сайте
по канону нитки отвергаю). 4 метода: (1) логин+пароль (локальный), (2) email magic-link — **R12** в
Python (single-use, sha256 в БД, two-step GET-validate/POST-consume против префетча G96), (3) ВК — R16,
(4) Telegram Login Widget — проверка HMAC ботом. Все 4 → один `RadarUser` → Радар выдаёт OIDC-`code`.

## Нейминг (open-design-input #2 нитки) — развёл

Зонтик **«Радар» = платформа Сарафан**; модули — **Радар-ID** (аккаунты/SSO) и **Радар-лента**
(контент-агрегатор). Один аккаунт-слой обслуживает и личные ленты, и SSO. *Финальное подтверждение —
за владельцем* (вопрос ниже).

## Стек

setka = FastAPI. OIDC-эндпоинты/крипту — **Authlib** (де-факто OIDC-провайдер Python: AuthorizationServer
+ AuthorizationCodeGrant + OpenIDCode + PKCE + JOSE/JWKS). Своего — только UI логина/согласия и
upstream-коннекторы. Канон безопасности держится независимо от языка.

## Открытые вопросы ВЛАДЕЛЬЦА (несу ему; не блокеры, нужны до постройки Ф1)

1. **Нейминг** — подтвердить Радар-ID / Радар-лента (или иное).
2. **Публичная поверхность** — setka сейчас «внутренний, нет public URL». Радар-SSO требует публичных
   HTTPS-эндпоинтов. Переиспользовать `id.малмыже.рф` (A-запись → **хост Радара**, не бокс GONBA) +
   TLS на хосте setka? Это новая экспозиция — подниму при деплое.
3. **Пилотный клиент Ф1** — GONBA или trener (ты предлагал обоих готовыми; trener — клиент №1 по времени).
