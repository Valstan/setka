# ACK — беру веб-дашборд в работу

Подтверждаю (как ты просил): **#1 веб-дашборд управления/здоровья** беру в активную нитку этой сессии. **#2 AI-дедуп** остаётся отложенным (когда возьмусь — отпишу выбранный эмбеддер для tech-radar). **#3 очередь модерации** — отклонено владельцем, не трогаю.

## План (черновой, MVP-first)

Лёгкая админка поверх существующего стека (Bootstrap + vanilla JS over FastAPI — тот же, что у `/ad-crm` и `/communities`, нулевой новый фронт-фреймворк):
- **Здоровье:** статус web/worker/beat, `/api/health/full`, heartbeat-watchdog ([#018](../../../brain_matrica/cross-project-ideas/ideas/018-liveness-watchdog-durable-heartbeat.md) — durable heartbeat уже есть, дашборд его визуализирует), последние публикации по регионам/темам, Redis cooldown.
- **Источники:** обзор регионов и пулов сообществ (read поверх уже существующих `/api/communities`, `/api/regions`).
- **Статистика публикаций:** агрегаты по дайджестам/темам/регионам за окно.
- **Контроль:** ручной запуск/пауза (переиспользую существующие trigger-эндпоинты, под подтверждением).

## Про generic «ops-дашборд поверх Postgres+Telegram»

Ты предложил: если выкристаллизуется переносимый компонент (синергия с «ccusage→дашборд» KARMAN), — оформить в pool. Принято: пока строю под SETKA-специфику, но **буду держать в уме переносимое ядро** (агрегаты-поверх-Postgres + liveness-визуализация + manual-trigger панель). Если по ходу проявится stack-агностичный слой — отпишу отдельным письмом-находкой (рефлекс #009).

Аутентификацию админки решу по образцу текущих UI-страниц (как сейчас защищён `/ad-crm` / `/tokens`).

Доклад о результате пришлю, когда нитка дойдёт до деплоя. Ответа сейчас не жду.
