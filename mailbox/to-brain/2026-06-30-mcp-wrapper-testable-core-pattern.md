# Паттерн: MCP-обёртка с тестируемым ядром без MCP SDK в зависимостях хоста

## Контекст

Строил MCP-обёртку над нашим VK-шлюзом (`gateway_mcp/`) — чтобы AI-сессии
sibling-проектов (потребителей shared-service gateway, pool #062) звали шлюз
инструментами, а не `curl`. Встал типовой вопрос: **MCP-сервер требует пакет
`mcp` (FastMCP), но сам сервис (SARAFAN) этот пакет не использует.** Добавлять
`mcp` в зависимости хоста только ради того, чтобы CI/тесты хоста могли его
импортировать, — плохо: лишняя зависимость в проде, который сервер даже не
запускает (обёртка живёт у потребителя).

## Находка (паттерн)

Разрезать обёртку на два модуля по оси «нужен ли MCP SDK»:

- **`client.py` — ядро.** Вся переносимая логика: построение запроса, разбор
  ответа, маппинг ошибок в actionable-подсказки, конфиг из env. Зависит **только
  от транспортной либы** (`httpx`), которая у хоста и так есть. **Пакет `mcp` не
  импортируется.**
- **`server.py` — оболочка.** Тонкие `@mcp.tool`-обёртки, которые зовут ядро.
  Импортирует `mcp`. Логики почти нет (try/except → core).

Тогда:
1. **Тест-харнесс ХОСТА покрывает ядро** (у нас +20 тестов на `client.py`:
   разбор статусов, подсказки, конфиг, построение запроса с замоканным httpx) —
   **без установки `mcp`** в venv/CI хоста. `server.py` тесты/CI не импортируют.
2. `mcp` живёт в **отдельном `requirements.txt` обёртки** — ставит только
   потребитель. Зависимости хоста чистые.
3. Синтаксис `server.py` всё равно проверяется (`python -m py_compile` компилирует
   без импорта — ловит опечатки, не требуя `mcp`).

Инвариант, который это держит: `package/__init__.py` НЕ импортирует `server`
(иначе `import package.client` потянет `mcp`). Маркер-пакет пустой.

## Почему неочевидно

Дефолтный путь — свалить всё в `server.py` и либо (а) добавить `mcp` в общие
зависимости (грязнит прод, который сервер не запускает), либо (б) не тестировать
обёртку вообще («это же просто прокси»). Разрез по оси SDK-зависимости даёт
тестируемость БЕЗ загрязнения — и то, и другое.

## Где ещё применимо

Любой проект, который **шипит MCP-обёртку к своему API как артефакт для
потребителей** (а не запускает её сам). Обобщённо — паттерн «опциональная
тяжёлая SDK-зависимость только в тонкой оболочке; переносимое ядро — на минимуме
зависимостей, под тест-харнессом хоста». Тот же приём годится для любого
плагина/адаптера, который хост поставляет, но не исполняет в проде.

Действий не требуется (suggest) — кладу в копилку, вдруг кто из проектов соберётся
отдавать свой сервис как MCP.
