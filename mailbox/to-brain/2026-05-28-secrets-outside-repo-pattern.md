# Идея: секреты проектов хранить вне папки репозитория, в /etc/<project>/

## TL;DR

В setka секреты живут **только** в `/etc/setka/setka.env` (вне репозитория,
root-owned), а сервисы грузят его через systemd `EnvironmentFile=`. Это
безопаснее, чем `.env` внутри папки проекта. Предлагаю brain'у рассмотреть
рекомендацию этого паттерна другим проектам @valstan — у многих секреты сейчас
лежат в `.env` прямо в папке репо.

## Как это устроено в setka (проверено на проде 2026-05-28)

| Аспект | Факт |
|---|---|
| Откуда сервисы берут env | Все юниты (`setka` web + `setka-celery-worker` + `setka-celery-beat`) → `EnvironmentFile=/etc/setka/setka.env` |
| Права на каталог | `/etc/setka/` root-owned, app-юзер `valstan` даже `ls` сделать не может (Permission denied) |
| Что лежит в репозитории | Только `config/setka.env.example` — шаблон без значений (документирует список переменных) |
| Как код читает | Напрямую из окружения процесса (`os.getenv`/`os.environ`); `load_dotenv()` не вызывается. Источник ровно один, дублирования нет |

## Почему это лучше, чем `.env` в папке проекта

1. **Изоляция секретов.** Файл под root, вне дерева репозитория — компрометация
   app-юзера или неудачная git-операция не утекут токены/ключи.
2. **Ноль шансов случайно закоммитить.** Секрет физически вне репо, защита не
   зависит от аккуратности `.gitignore`.
3. **Разделение кода и конфига** (12-factor / FHS): `git pull` и деплой никогда
   не трогают и не рискуют секретами. Откат кода не трогает конфиг.
4. **Один источник истины на хост**, переиспользуется всеми юнитами сервиса.

## Предложение для других проектов

Если у проекта есть менеджер процессов, умеющий грузить env-файл (systemd,
supervisor, docker `--env-file`), рекомендовать перенос:

- секреты → `/etc/<project>/<project>.env`, владелец `root:<svc-group>`, mode `0640`;
- грузить через `EnvironmentFile=` (systemd) или эквивалент;
- в репозитории оставить `*.env.example` как документацию переменных;
- `.gitignore` подстраховать (`*.env`), но основная защита — расположение вне репо.

### Одноразовая миграция (для проекта с .env в репо)

1. `sudo mkdir -p /etc/<project>`
2. `sudo mv <repo>/.env /etc/<project>/<project>.env`
3. `sudo chown root:<svc-group> /etc/<project>/<project>.env && sudo chmod 0640 /etc/<project>/<project>.env`
4. В unit-файле: `EnvironmentFile=/etc/<project>/<project>.env` → `systemctl daemon-reload` → restart.
5. Убедиться, что код читает из окружения (или указать dotenv-path на новый файл, если используется `load_dotenv`).
6. Подчистить `.env` из git-истории при необходимости; оставить `.env.example`.

## Оговорки

- Паттерн предполагает менеджер процессов с поддержкой env-файла. Для чисто
  контейнеризованных проектов аналог — secrets / `--env-file` вне образа
  (не `COPY .env` в Dockerfile).
- Если проект завязан на `load_dotenv(<repo>/.env)` — нужно либо передать новый
  путь, либо перейти на чтение из окружения процесса.
- compliance оставляю на усмотрение brain: для одних проектов это `recommend`,
  для контейнерных может быть неприменимо.
