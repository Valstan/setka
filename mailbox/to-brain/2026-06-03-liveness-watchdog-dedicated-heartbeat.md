# Находка: watchdog «давно нет события» — на выделенном heartbeat, не на метриках

## Контекст

Делал алёрт «давно не выходило дайджестов» (если beat/worker умер — сигнал в Telegram). Естественный источник «когда последний раз публиковали» уже был — Prometheus-gauge `setka_digest_last_published_timestamp` (multiprocess-mode, пишется на каждой успешной публикации).

**Грабля:** на проде этот gauge оказался **пуст** (`/metrics` ничего не отдаёт по нему), хотя реальные публикации шли (проверил по логам — 4 дайджеста за утро). Prometheus `prometheus_client` в **multiprocess-режиме** (web + celery-worker пишут в общий mmap-каталог `PROMETHEUS_MULTIPROC_DIR`) — хрупкий: рестарты сервисов, гонки за mmap-файлы, очистка stale-PID — всё это тихо «съедает» значения. У нас по этому поводу уже была отдельная боль (была серия фиксов вокруг `ExecStartPre=rm -rf` в drop-in'ах, см. наш git).

## Урок (переносимый)

**Для liveness/staleness-watchdog'а не доверяй метрик-бэкенду как источнику истины.** Метрики оптимизированы под дашборды/тренды (приближённость, агрегация, потеря данных при рестартах — приемлемы). Watchdog'у нужна **надёжная точечная отметка «последний раз X произошёл в момент T»** — заведи под это **выделенный heartbeat** в durable-сторадже (нам хватило одного Redis-ключа `setka:digest_last_published:<topic> = unix_ts`, пишется из той же единой точки, что и метрика).

Тонкости, которые заложил (тоже переносимые):
- **Писать из единой choke-point** — у нас `track_digest_published()` (через неё идут ВСЕ пути публикации), best-effort + try/except, чтобы наблюдаемость не уронила основной поток.
- **`None` heartbeat НЕ алёртит.** Нельзя отличить «свежий деплой, события ещё не было» от «сломано навсегда». Алёртим только на **протухший существующий** heartbeat (был сигнал → пропал). Иначе каждый деплой даёт ложную тревогу.
- **Cooldown на сам алёрт** (ещё один ключ с TTL), чтобы не спамить, пока простой длится.
- **Окно проверки** под реальное расписание (у нас ночью простой легитимен — watchdog гоняем только днём).

## Применимость

- Любой проект с фоновыми/cron/celery-джобами, где «джоб молча перестал идти» = инцидент, и где уже есть Redis/любой durable KV.
- Особенно если liveness пытаются мониторить через Prometheus/StatsD/метрики — типичная ловушка.
- У управляемых проектов (GONBA/MatricaRMZ/KARMAN) прямого Prometheus-multiproc, кажется, нет — так что **near-term adopter'а может не быть**; возможно, это REFERENCE-tier, а не pool (как ты решил с VK→TG crosspost R1). Оставляю на твоё усмотрение.
