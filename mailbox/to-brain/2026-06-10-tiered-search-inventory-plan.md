# #035 — инвентарь, план, вопросы (setka)

## 1. Инвентарь полей поиска (#022, полный sweep web/templates + web/static/js + web/api)

**Свободный текст (кандидаты на tiered) — всего 3:**

| Поле | Где матчится | Сейчас |
|---|---|---|
| `/posts` `#filter-search` (содержимое постов) | клиент, JS | `.includes()` — substring, Ур.1 уже есть |
| `/subscriber-growth` `#growth-search` (названия регионов) | клиент, JS | `.includes()` — substring |
| `/ad-crm` `#filter-q` (name/contact клиентов) | сервер, `web/api/ad_crm.py` | `ILIKE '%q%'` — substring |

**Неприменимо:** typeahead городов (`/regions/new`, `/regions/<code>/prepare`) матчит **VK API** `database.getCities` — ранжирование на их стороне, наш модуль не воткнуть (разве что переранжировать выдачу — мелочь, в Ф3+). ~15 остальных фильтров — exact-enum дропдауны (регион/категория/статус/стадия), tiered не нужен. Ожидаемого «поиска по рекламному кабинету» нет: `/ad-cabinet` фильтруется только enum'ами (origin/status), текстового поля там нет.

**Главная находка инвентаря:** на `/communities` — самой большой операторской таблице (~840 сообществ) — текстового поиска **нет вообще**, только дропдауны. Оператор ищет паблик скроллом. Добавить поле с tiered-матчем по name/screen_name = бóльшая часть пользы директивы.

## 2. Фазовый план

- **Ф0** — shared-модуль `web/static/js/search_match.js` по спеку #035: нормализация (lower, `ё→е`, схлопывание пробелов, compact-номера) → substring → subsequence → fuzzy (чистый JS, без зависимостей) → многотокен AND → ранжирование (exact > prefix > word-prefix > substring > subsequence > fuzzy) → подсветка → RU↔EN ретрай на нуле результатов.
- **Ф1 (пилот)** — `/subscriber-growth` + `/posts`: списки уже в памяти клиента, подключение дешёвое, риск нулевой.
- **Ф2** — `/communities`: новое поле поиска (та же клиентская фильтрация загруженного списка; 840 строк для JS — ничто).
- **Ф3** — серверный `/ad-crm?q=`: нормализация запроса + при 0 результатов fuzzy-фолбэк `pg_trgm similarity` (миграция `CREATE EXTENSION pg_trgm` + GIN) + RU↔EN ретрай. Probe (#020): сейчас `ad_clients` — десятки строк, латентность не вопрос; GIN-индекс заведём сразу, чтобы не вспоминать при росте. RAM-цена pg_trgm на нашем tiny-VPS ничтожна (индекс на килобайтных таблицах).

Статус: план в `docs/PENDING_FOLLOWUPS.md` (⏳, `⏱ 2026-06-10 · fresh`), реализация — после OK владельца по приоритету.

2. Подсветка — везде или только в выпадашках/typeahead (MatricaRMZ-решение: выпадашки + где дёшево)?
3. Приоритет нитки относительно очереди освежения регионов (`nolinsk` сверху).

## 4. Находки (#009)

- Перенос для pool #035: паттерн «инвентарь сначала» снова окупился — интуитивный объём («поиск везде») против фактического: 3 поля, из них 2 уже substring. Реальная дельта директивы в setka = subsequence/fuzzy/раскладка на 3 полях + **одно отсутствующее поле**, а не тотальный рефакторинг. Совпадает с опытом MatricaRMZ («~70% уже построено»).
- `/posts` `#filter-search` фильтрует только **загруженную страницу** выдачи (бэкенд `web/api/posts.py` параметра поиска не имеет) — оператор думает, что ищет по всем постам, а ищет по 50 видимым. Это не #035-дефект, а скрытый UX-обман; занесли как кандидата на серверный `?q=` в Ф3+.
