# Два паттерна работы с облачными рутинами-агентами

Контекст: setka использует облачную рутину (claude.ai/code Routine) как «мотор»
HITL-классификатора постов (нет `ANTHROPIC_API_KEY` → классификацию делает
scheduled cloud agent через HTTP-интерфейс к setka). За несколько сессий
накопились два решения, переносимых на любой проект экосистемы, который
запускает облачные рутины-агенты против своего бэкенда.

## Паттерн 1. Рутину нельзя настроить / она отбивается → верни задачу в dev-чат как on-demand skill

**Боль.** Облачная рутина хрупка по двум осям: (а) её конфиг (модель, промпт,
network) правится только через web-UI claude.ai, и нужную настройку не всегда
удаётся найти/применить; (б) модель рутины периодически over-refuse'ит
автономную задачу (читает «фоновое событие, а не сообщение пользователя» →
«standing by», либо авторизационный контекст в промпте → «social engineering»).

**Решение.** Не каждую периодическую AI-задачу обязательно держать облачной
рутиной 24/7. Если задача (а) идёт редко (раз в день/неделю), (б) требует
human-in-the-loop и так, (в) капризна в настройке — **дешевле вернуть её в
dev-чат как on-demand skill с человеческим триггером**: вся HTTP-обвязка в
детерминированном скрипте (ключ читается с VPS, не покидает прод), модель в
чате делает содержательную часть по запросу, результат идёт в тот же
human-review-контур. Плюс журнал запусков + напоминание в `/start`, если давно
не запускали. У setka так заменили рутину-дистиллятор правил на скилл `/distill`.

**Эвристика выбора:** облачная рутина оправдана для частых автономных задач без
человека в петле (наш классификатор — ежечасный, остаётся рутиной). Редкая +
human-gated + капризная → dev-чат скилл.

## Паттерн 2. Media/данные из-за egress-firewall облака → прокси на своём allow-listed хосте

**Боль.** Cloud environment рутины по умолчанию не пускает произвольный исходящий
трафик (Network access: Trusted/Custom — только явный allowlist хостов). Рутине
нужно посмотреть картинки/PDF, которые лежат на чужом CDN (у нас — VK
`*.userapi.com`), но добавить пол-интернета в allowlist нельзя, а VK ещё и
привязывает выдачу к IP.

**Решение.** Один **media-прокси-эндпоинт на своём (уже allow-listed) хосте**:
рутина просит ресурс через `GET /api/.../media?url=...` со своим API-ключом,
бэкенд скачивает с внешнего CDN и отдаёт байты. Ключевое — прокси **не открытый
SSRF-фетчер**: allowlist хостов-суффиксов источника, только https, потолок
размера, повторная проверка финального URL после редиректов. Так облачный агент
за egress-firewall получает доступ ровно к нужному классу ресурсов через одну
дверь, которую он и так может открыть.
