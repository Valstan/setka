# Находка: HITL-петля «человек ревьюит вердикты машины» — два неочевидных паттерна

Углубляя shadow-классификатор SETKA (оператор размечает вердикты нейронки, копится agree-rate), напоролся на два дефекта, которые, кажется, легко повторить в любом human-in-the-loop / annotation / content-moderation проекте экосистемы. Оба — про **измерение согласия** и **слепые зоны ревью**. Кладу в pool на случай, если у кого-то похожая петля (или будет — trener, будущие модерации).

## Паттерн 1: «есть реакция» ≠ «вердикт завершён» + «правка, совпавшая с ИИ» = согласие

**Боль.** Лента ревью прятала элемент после ЛЮБОГО клика (`only_unreacted = есть хоть одна коррекция`). Два зла сразу:
1. **Составной вердикт невозможен** — оператор хотел сменить тему И действие, но карточка исчезала после первого клика.
2. **Метрика испорчена** — оператор жал кнопку «→ публиковать» на посте, где ИИ уже поставил publish, чтобы ПОДТВЕРДИТЬ; система писала это как «коррекцию» (несогласие). Из 19 «коррекций действия» 9 были `operator_value == ai_value` → agree-rate искусственно занижен (60% вместо реальных 77%).

**Лечение (обобщённо):**
- Разделить **«реакция»** и **«финализация»**: элемент уходит из очереди только по явному «готово», не по факту любой реакции. Так возможен многошаговый вердикт.
- В записи реакции: если `operator_value == ai_value` — писать **agree**, а не coррекцию. Кнопка, совпавшая с предсказанием модели, — это согласие, не правка. Иначе метрика «доля согласия» врёт вниз, и measure-before-enforce гейт не откроется никогда.

## Паттерн 2: аудитить ОБЕ стороны детерминированного фильтра, не только «прошедшее»

**Боль.** LLM-классификатор в shadow читал только то, что фильтр пропустил И что опубликовалось. Значит он ловил только **пере-публикацию** (мусор, который просочился), но был **слеп к пере-фильтрации** — хорошим постам, которые детерминированный фильтр выбросил зря (они нигде не сохранялись, отсев шёл в памяти).

**Лечение (обобщённо):** shadow-журнал фиксирует КАЖДЫЙ вход с решением фильтра — `kept` | `dropped`+причина. Тогда человек/LLM видят обе стороны: и «зря пропустили», и «зря выбросили». Это замыкает петлю калибровки фильтра по данным, а не по интуиции. Захват — fail-safe на границе (не трогая логику самого фильтра: изолированная сессия, never-raises, gated), причину отсева пере-выводить теми же чистыми функциями. Механические отсевы (дедуп/возраст) не писать — только «суждение», иначе шум затопит ревью.

## Почему в pool

Обе штуки **не про домен setka** — это про конструкцию любой петли «модель предлагает → человек подтверждает/правит → метрика согласия → enforce по порогу». Грабли неочевидные: инстинкт — прятать разобранное и ревьюить только прошедшее; оба инстинкта тихо ломают именно то, ради чего петля затевалась (честная метрика + полнота обзора). Дизайн у нас — ADR-0003/0004 в репо setka (читаемы напрямую по ADR-0007, если пригодится).
