# Сетевая рассылка — probe-ответ и план (probe-before-build, #020)

Отвечаю на директиву `2026-06-14-network-broadcast-internal-scheduler.md`. Probe не строил с
нуля — он **уже сделан вживую** потоком «Кругозор» (научпоп-дайджест веером на всю сеть,
работает на проде с 2026-06-14). Цифры ниже — не оценка, а факт прода.

## Probe п.1 — wall.post-права по сети (охват)

**Охват = 16 пабликов.** Это все активные регионы с привязанным `vk_group_id`
(`SELECT count(*) FROM regions WHERE is_active AND vk_group_id IS NOT NULL AND code<>'copy'`
= 16) — районные + областные **главные инфо-группы** сети. Krugozor публикует веером ровно
на этот набор → эмпирически **16/16 `wall.post` успешно** (live-прогон верифицирован).

**Тип токена — гибрид (publish-полиси уже решает это сама):** `VKPublisher._client_for_group`
для каждой цели берёт **community-токен**, если он есть в роутинге для этой группы, иначе —
общий **user-токен** publish-кандидата. То есть постинг идёт под тем токеном, под которым у
нас реально есть права на конкретный паблик. Это снимает G19-развилку: не «один тип токена на
всё», а per-target выбор. **Рассылке отдельный probe прав не нужен — переиспользуем тот же
`create_with_policy`-стек, что и krugozor.**

**Граница охвата:** «все паблики области» = эти 16 (где мы админ/есть токен), **не** произвольные
чужие сообщества. За пределами 16 — `[15] Access denied` (не админ). Если потом сеть авто-регистрации
вырастет — охват вырастет автоматически (цель = активный регион с vk_group_id, как у krugozor).

## Probe п.2 — антиспам/флуд-лимиты (throttle)

- **16 `wall.post` с интервалом 5 с = 16/16 без капчи** (live krugozor; дефолт
  `KRUGOZOR_POST_INTERVAL_SECONDS=5`, «как у copy_setka»). Полный веер 16 целей @5с ≈ **80 с** —
  комфортно для разовой кампании.
- **Бурст роняет в капчу:** 16 `wall.edit` подряд @3 с словил VK-Captcha (как и copy_setka на
  бурсте). → **Правило: throttle ≥5 с между публикациями, без бурстов.** Это и есть throttle беата
  рассылки — параметризуемый интервал, дефолт 5 с.

## Probe п.3 — повтор одинакового контента

Honest-caveat: krugozor льёт **РАЗНЫЙ** дайджест per-run (ротация источников), поэтому
идентичный-текст-на-много-целей живьём **не** проверял. Риск VK dup-detection при рассылке
**одного и того же** поста на 16 пабликов выше, чем у разнящегося дайджеста. Закладываю:
1. интервал ≥5 с между целями (см. п.2) — уже снижает паттерн «спам-волна»;
2. **опц. лёгкая вариация per-target** (микро-отличие: эмодзи/перестановка/хвост) — hook в
   модели, **off по умолчанию**, включаем если живьём увидим дубль-флаги. Не лупим один текст
   подряд без паузы.

## Канон подтверждён — свой планировщик, не VK-отложка

Подтверждаю реализуемость канона владельца. Публикатор **умеет** `publish_date` (кладёт в
VK «Отложенные») — но по решению владельца **мы это не используем** для рассылки: расписание +
повтор + очередь держим **в своей БД**, beat публикует `wall.post` **немедленно** в момент
срабатывания. Это ровно снимает боль владельца (правка расписания/текста — в одном интерфейсе
SARAFAN, а не лазить по сотням отложек VK).

## Переиспользование ad-CRM С2 (не дублируем планировщик)

Как и просили — не строю второй планировщик. ad-CRM С2 (`modules/ad_cabinet/post_expirer.py` +
beat `expire-ad-posts-daily`) — это уже «очередь действий по дате + beat + per-target исполнение
через VK API». Рассылка = обобщение того же контура: с «снять пост по дате» на «опубликовать
(и повторить) по расписанию в N целей». Те же примитивы:
- **сущность «кампания»** (контент + цели + расписание + повтор), редактируемая до/между публикациями;
- **per-target статус-защёлка** (`queued/published/error+reason`) → идемпотентность (повторный беат
  не публикует дважды в одну цель);
- **per-target изоляция** (ошибка в один паблик не валит остальные — как krugozor `except` per-region);
- **watchdog #018**: «беат рассылки молча умер» ≠ «нечего слать» — durable heartbeat, как на мониторинге.

## Итог / следующий шаг

Охват реальный = **16 пабликов** (вся текущая сеть), throttle = **≥5 с**, дубль-вариация —
заложу hook. Канон и переиспользование подтверждены живыми данными — **probe-гейт пройден, строить
не вслепую.** Если 16 как MVP-охват ОК — следующим ходом проектирую модель данных кампании + UI
очереди под #027 (тактика моя), PR-флоу. Если ждёшь что-то ещё до стройки — верни письмом.
