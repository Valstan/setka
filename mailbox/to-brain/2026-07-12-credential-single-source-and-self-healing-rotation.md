# Находка: устойчивая ротация API-креденшелов (single-source + probe + self-heal)

## Боль (реальный инцидент SETKA 2026-07-12)

Автопубликация районных сводок молча встала на **4 дня**. Корень — VK user-токен
был **аннулирован провайдером** (аккаунт временно забанен), но:

1. **Хранимый флаг `validation_status='valid'` устарел** — в UI/БД токен числился
   живым, хотя live-вызов `users.get` уже отдавал «invalid». Флаг ставился при
   прошлой валидации и никто его не пересматривал.
2. **Горячий путь чтения брал `next(iter(tokens))` — один токен, без перебора и без
   auto-disable на ошибку.** Мёртвый токен стоял первым → каждый вызов падал → 0
   данных → 0 публикаций. Живой второй токен (был!) в очередь не доходил.
3. **Рассинхрон двух источников токена.** Значение жило и в БД (читал парсинг), и в
   env-файле (читали остальные модули). Владелец обновил только UI/БД → парсинг ожил,
   а env-пути ещё сутки висли на старом токене. «Обновил токен» ≠ «обновил везде».

Дополнительная грабля-провайдер: **разблокировка аккаунта ≠ восстановление токена** —
строку токена провайдер аннулирует безвозвратно, нужен новый OAuth. И **`join`/
`editManager`-подобные write-методы часто требуют browser-validation** (`error 17`)
для не-свежего токена — свежий проходит, старый нет.

## Паттерн (что построили, переносимо)

Три принципа, применимы к **любому** проекту с пулом ротируемых API-креденшелов
(VK, любой внешний API, OIDC-клиент-секреты, шлюз с несколькими ключами):

1. **Единый источник правды для креденшела.** Одно место записи (у нас — БД через
   UI), остальные конфиги — максимум bootstrap-fallback, который НЕ может воскресить
   выключенный креденшел. Убирает класс багов «обновил в одном месте из двух».
2. **Probe-before-use вместо доверия хранимому флагу.** Перед горячим использованием —
   дешёвый live-probe (`users.get`/`whoami`). Мёртвый (по коду ошибки) уходит в
   **cooldown автоматически** (+alert), берётся следующий кандидат. Сетевой сбой probe
   ≠ вина креденшела (не дисквалифицируем). Флагу в БД не доверяем как «жив».
3. **No-single-pick: перебор + карусель.** Горячий путь перебирает кандидатов, а не
   берёт первого. Бонус — распределение нагрузки: отдавать по `last_used ASC` и штамповать
   при выборе (равномерно по всему пулу, а не долбить один ключ до rate-limit).

Публикация/запись при этом — **каскад эшелонов** (у нас: токен-сообщества → основной
user-токен → резервный user-токен), где резерв включается только при отказе старших.

## Кому может пригодиться

Любой sibling с внешними API и ротацией ключей: VK-шлюз-потребители (малмыжские сайты,
GONBA/Sabantuy), Радар-ID клиент-секреты, будущие интеграции. Особенно там, где «приём/
публикация молча встала» дорого (перекликается с watchdog-темой [#018]).

Если решите поднять в pool/REFERENCE — ключевой одноразовый урок:
**«не доверяй хранимому флагу валидности креденшела — probe живьём; и один
мёртвый-но-включённый ключ в no-fallback пути кладёт весь пайплайн молча».**
