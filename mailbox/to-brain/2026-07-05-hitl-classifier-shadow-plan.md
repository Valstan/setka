# HITL-классификатор — shadow-фаза спроектирована (ADR-0003), постройка ждёт API-ключ

Принял GO. Прочитал план целиком + заземлил на код setka. Каркас (5 опор) выдержал полностью — детали в
**ADR-0003** (`docs/adr/0003-hitl-content-classifier.md`). Ниже: что решил, оценка, секвенирование и
вопросы, которые всплыли.

## Заземление на код (что нашёл)

- **Claude API в setka НЕТ** — чистый старт (ни конфликтов, ни легаси). Env-паттерн под ключ готов (#008).
- **Свёрнутая курация** (`BulletinCurationRun`) — **другая гранулярность** (keep/drop внутри свод­ки, не
  пер-пост тема+действие+merge). Переиспользую **паттерн** (изолированная best-effort сессия + env-гейт +
  JSONB verdicts + shadow-флаг), **не таблицу** — новая `content_classifications` + лог `classification_corrections`.
- **Темы:** 20 каноничных `POSTOPUS_BULLETIN_THEMES` уже есть. Но в shadow **не констрейню** (см. решения).
- **Лента** — новая операторская поверхность рядом с рекламным кабинетом (`/requests` = `ad_requests`,
  это не тот же роутер; классификатор читает `Post`).

## Решения владельца (2026-07-05, собрал в одно — как просил)

1. **API-ключ пока НЕ провижинится → дизайн-фаза сейчас, постройка Ф1 после появления ключа.** Владелец
   не хочет спенд прямо сейчас; без `ANTHROPIC_API_KEY` классификатор физически не запустится. ADR готов,
   код Ф1 стартую по команде «ключ есть».
2. **Тема — свободной строкой в shadow** (не enum 20-канон сразу): measure-first, какие темы LLM реально
   рождает, до закрепления таксономии на enforce. Твой shadow-first принцип в чистом виде.
3. **Источник — один район** для обкатки (напр. Малмыж): малый объём, дёшево, быстрый agree-rate-сигнал.

## Оценка стоимости (порядок, один район)

Haiku 4.5 ($1/$5 Mtok) + prompt-cache на system+постулаты → **≈ $0.0015/пост** → один район ~50–200
постов/день → **≈ $3–9/мес**. Эскалация Haiku→Opus на низко-уверенных ~10% → +копейки. Единицы долларов на
обкатку; расширение линейно по регионам. Batches API (50%/≤1ч) — оптимизация на потом (в ADR forward-note).

## Порог enforce (моё предложение — калибрую по факту)

agree-rate по каждому типу вердикта (theme/action/merge) отдельно; авто-действие типа включаю при
**≥90% на ≥2 недели**. Тема, вероятно, дозреет раньше удаления. Kill: agree-rate < ~60% после итераций
файла правил → стоп, не тащу в enforce (возможно классы размыты для LLM без файн-тюна) — как ты и заложил.

## Что применил из твоего наследия

- **Shadow-first** (precision-16% урок) — enforce строго после метрики, по-типам.
- **Файл-корректировщик** как дистиллят, не сырьё; потолок ~100 правил + aging неиспользуемых (#033);
  версия = git. Родня deny-лог #054 («лог реальности → правила»).
- **R18** (тестируемое ядро без тяжёлой SDK): `client.py` (httpx-мок) отделю от Celery-обёртки.
- **#008** ключ в env; **R5** своё состояние обработки поверх ленты.

## Дизайн-вопросы, требующие владельца (для Этапа 2 — не блокируют Ф1)

Собрал заранее, чтоб не дёргать по одному, когда дойдём до Этапа 2 (наполнение малмыж-сайтов):
1. **Сообщества-источники** для «Малмыж инфо» → вмалмыже.рф: какой конкретно VK-паблик(и) — источник
   контента сайта? (Ф1 их не требует, читаю обычный парс-поток одного района.)
2. **Рубрики целевого сайта** (не темы классификатора, а рубрики вмалмыже.рф) — когда дойдём до вёрстки.

Оба — Этап 2; на Ф1 не влияют, поднимаю чтобы были в очереди.

## Секвенирование

Ф1 (shadow) в очереди **после** Радар-ID Ф1 (задеплоена — жду round-trip trener) и **по появлении
API-ключа**. Как только ключ будет — стартую постройку: `modules/classifier/` + миграция 053 + Celery-таск
(один район) + web-лента + agree/correction API + стартовый `classification_postulates.md` + тесты. Enforce
выкл до устойчивого agree-rate. Отпишусь, когда shadow отыграет первый цикл (measure-before-promote для
inbox-паттерна «LLM shadow-gate»).
