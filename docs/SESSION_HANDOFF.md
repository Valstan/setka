**Что сделано сегодня (PR [#262](https://github.com/Valstan/setka/pull/262), смержен + задеплоен):**
- **413 при загрузке картинки >1 МБ — ИСПРАВЛЕНО.** Корень оказался нетривиальным (две итерации): хостер myjino обрывает HTTPS на **своём edge-прокси** и шлёт на nginx VPS по **порту 80** с `X-Forwarded-Proto: https` → пользователь попадает в **Block 3** (`listen 80`, `server_name 3931b3fe50ab…`, файл `setka_http_redirect_domain.conf`), а НЕ в 443-блок. У Block 3 не задан `client_max_body_size` → дефолт 1 МБ. **Фикс: `client_max_body_size 20m` на уровне `http` в `/etc/nginx/nginx.conf`** — наследуют все блоки. Проверено по реальному пути (nginx:80 + Host + XFP=https): было 2 МБ→413, стало 2 МБ/8 МБ→404 (лимит снят).
- **Удаление картинок — UX.** Кнопка-«×» была почти невидимой ссылкой; сделал явную красную круглую кнопку + бейдж «в посте» + поясняющий текст. Бэкенд `DELETE /api/broadcast/images/{name}` уже был.
- **Кликабельность ссылок.** VK сам линкует `https://…` в `message` (`publish_digest` кладёт текст в `message` как есть) — код не трогал, добавил подсказку под полем текста.

## Failed approaches (этой нитки)

- **Поднять `client_max_body_size` только на 443-блоке (`conf.d/setka.conf`)** — не помогло: внешний трафик идёт через edge-прокси myjino на **nginx:80 (Block 3)**, а не на локальный 443. Фикс обязан быть на уровне `http` (или на :80-блоке). **Урок:** при провайдерском HTTPS-edge перед бэкенд-nginx правь лимиты на том блоке, что реально слушает edge (обычно :80), либо глобально в `http`. Диагностика — `nginx -T` (видеть ВСЕ server-блоки) + тест по реальному пути `curl :80 -H "Host: …" -H "X-Forwarded-Proto: https"`.
- **Гипотеза «виноват Starlette 0.48 `max_part_size`=1 МБ»** — опровергнута локальным репро: `UploadFile = File(...)` спокойно принял 3 МБ (и старый, и новый паттерн → 200). Приложение/VK лимит не ставят; прикладной лимит broadcast — 12 МБ (отдаёт 400, не 413).
- **Диагностика 413 неаутентифицированным curl через nginx** — врёт: AuthGate отвечает 302/401 не дочитав тело multipart → nginx видит «битый» upstream → **502** (артефакт, не настоящий код). Мерить лимит тела на статик-пути (`/.well-known/acme-challenge/…`, без app/auth) — там nginx режет тело ДО обработки, чисто 413 vs 404.
