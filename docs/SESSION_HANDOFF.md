## Текущая нитка

**Рекламный кабинет 2.0.** Блоки **B1** (планировщик отложенных постов) и **B2** (запланировать рекламную заявку из предложки) — реализованы и задеплоены. Остаются блоки **A** (реклама во входящих ЛС) и **C** (учёт оплат/публикаций, CRM) + owner-браузер-верификации.

**Главный вывод сессии (B2):** исходный замысел «in-place `wall.edit` предложки → отложка» **технически невозможен** (живой VK-probe) — предложку правит только web-UI VK, не API. B2 развёрнут на **пересоздание через движок B1**: «Запланировать» на заявке → `wall.post(publish_date)` создаёт новый отложенный пост, оригинал убирается из предложки (`wall.delete`), заявка → `published`. Родная подпись «Предложил(а): автор» при этом не сохраняется (честно отмечено в UI).

## Следующий шаг

Приоритет сверху вниз (на выбор владельца):

1. **Браузер-верификация B2** (owner-шаг, агент UI не открывает): `/ad-cabinet` → «Запланировать» на реальной заявке → задать дату (+10–15 мин) → «Отправить в сообщество» → проверить: пост появился в VK-«Отложенных записях» группы, оригинал ушёл из предложки, заявка стала «Опубликовано». Тумблер «убрать оригинал» вкл. по умолчанию. При проблемах — `ssh setka "journalctl -u setka --since '5 min ago' --no-pager | tail -50"` (composer публикует синхронно через web).
2. **Блок A — реклама во входящих ЛС**: beat-детект в `messages.getConversations` → `ad_requests.origin='inbound_dm'` (новая колонка/миграция); тред-вью `messages.getHistory` (добавить в `VKClient`) + ответ из кабинета (готовый `send_message`). API-выполнимо.
3. **Блок C — учёт оплат/публикаций** (CRM): `ad_clients`/`ad_payments`/`ad_publications` (ключ `author_vk_id`), воронка detected→contacted→scheduled→published→paid. Задел: `ad_scheduled_posts.client_id`/`price` (025, nullable, без FK).
4. **Браузер-верификация B1** (если ещё не делалась владельцем): создать отложенный пост из composer'а → проверить в VK-«Отложенных».

## Не забыть (low-priority)

- 🟢 **Медиа автора предложки при пересоздании B2 не переприкрепляется** (`photo{user}_{id}` — чужое фото, доступ) — оператор выбирает офферные картинки / грузит свои. Если понадобится — скачивать+перезаливать оригинальные фото (как `vk_wall_photo_upload`). См. `PENDING_FOLLOWUPS.md`.
- ℹ️ **B1 rate-limit:** N отложенных постов в одну группу подряд тормозят на `POST_INTERVAL_SECONDS=5` (5 дат ≈20с синхронно в web-запросе). Приемлемо для MVP; при жалобах — вынести в Celery-фон.
- ℹ️ Находка «`wall.edit` недоступен для предложки» отправлена в мозг (`mailbox/to-brain/2026-06-04-vk-probe-before-build.md`) как кандидат в GOTCHAS/паттерн «probe-before-build».
