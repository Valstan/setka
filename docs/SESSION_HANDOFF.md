## Текущая нитка

Большая сессия по HITL-классификатору: разобрали, как он реально работает, и углубили на трёх фронтах — **всё построено, смёржено и задеплоено**. Осталось накопление (оператор + аудит) и настройка облачной рутины.

1. **Лента вердиктов ([#312](https://github.com/Valstan/setka/pull/312)):** модель «черновик → финализация» — пост уходит из ленты только по явному «✅ Согласен со всем» / «✔ Готово» (`reviewed_at`), составные вердикты возможны. Клик, совпавший с ИИ, пишется как agree, не ложная коррекция → agree-rate честный (action на проде 60%→**77%** после бэкфилла). Файл-постулаты обучен градацией коммерции + чужой район.
2. **Аудит обеих сторон ([#313](https://github.com/Valstan/setka/pull/313), ADR-0004):** находка — классификатор читал `bulletin_curation_runs.candidates` = **только опубликованное** → был слеп к пере-фильтрации. Новый `collected_post_audit` фиксирует каждый собранный пост с решением фильтра (kept | dropped+причина); классификатор видит обе стороны. Захват fail-safe на границе сбора (`_filter_post` не тронут), gated env.
3. **Детерминированный отсев коммерции ([#314](https://github.com/Valstan/setka/pull/314)):** `is_advertisement` теперь ловит «продам/продаю/куплю/обменяю» → частная продажа товара режется до публикации (услуги мастеров не трогаются).

## Следующий шаг

Кода не требуют — за оператором и накоплением:

1. **Оператор:** разобрать в `/classifier` (меню «Система» → Классификатор) **8 свежих вердиктов**, засеянных в конце сессии (ручной прогон рутины с прод-стороны) — заодно обкатать новый флоу «Готово»/составные правки. Копится agree-rate.
2. **Облачная рутина (№2):** локальная задача `setka-hitl-classifier` создана на ЭТОЙ машине (раздел «Scheduled» в сайдбаре Claude Desktop, cron `0 */3 * * *`). Чтобы заработала: (а) вписать `CLASSIFIER_INGEST_KEY` в `C:\Users\valstan\.claude\scheduled-tasks\setka-hitl-classifier\SKILL.md` (взять `ssh setka 'sudo grep CLASSIFIER_INGEST_KEY /etc/setka/setka.env'`); (б) нажать «Run now» (одобрит сетевые инструменты). Крутится, пока приложение открыто. Истинный 24/7-сервер — отдельный продукт, отсюда не провижинится.
3. **Проверить аудит после сбора:** `collected_post_audit` пуст, наполнится при следующем сборе `mi` (beat). Тогда в ленте появятся бейджи «🚫 отсеян / ✅ прошёл фильтр» — глянуть, нет ли over-drop.

## Failed approaches (этой сессии)

- **Bare `./venv/bin/python -c` на проде без загрузки env** — `config.runtime` требует `DATABASE_URL` на импорте, systemd грузит env сам, а голый python — нет → `RuntimeError: DATABASE_URL is required`. Запускать прод-python с env: `sudo bash -c 'set -a; source /etc/setka/setka.env; set +a; ./venv/bin/python ...'` (память [[curate-prod-env-sourcing]]).
- **Инлайн-python через `chr()`-эскейпинг в ssh→sudo→python** — легко передать не ту строку (передал `"mi"` с кавычками вместо `mi`). Для нетривиальных прод-проверок — писать во временный `.py` через heredoc, потом запускать.
- **`create_scheduled_task` (scheduled-tasks MCP) ≠ облачный 24/7** — создаёт ЛОКАЛЬНУЮ задачу Claude Desktop («крутится, пока приложение открыто»), не серверную. Истинный claude.ai/code-cloud отсюда не завести.

## Открытые вопросы для пользователя

- **is_advertisement глобальна (#314):** отсев «продам/куплю» затрагивает ВСЕ районы. Если сельский район захочет оставлять «продам корову» — откат к per-region через `RegionConfig.delete_msg_blacklist`. Пока аудит есть только для `mi` → over-drop в других районах не виден.
