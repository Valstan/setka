**Починка «Сетевой рассылки»: картинки терялись (запрос владельца 2026-06-19)** — найдено, пофикшено, задеплоено за сессию, осталась браузер-проверка владельцем. Параллельно живы standing-нитки прошлых сессий (ждут владельца): спот-чек 19 LLM-флагов, каркас обложек, «Радиоточка» (TG/VK-личка).

**Что сделано сегодня (PR [#264](https://github.com/Valstan/setka/pull/264), смержен + задеплоен, прод HEAD `1925005`):**
- **Рассылка слала только текст, картинки терялись — ИСПРАВЛЕНО.** Корень (лог worker): диспетчер грузил фото на стену **community-токеном** → VK `[27] method is unavailable with group auth` (тот же барьер #27, что у `wall.edit`/`stats.get`) → `upload_wall_photo` возвращал None по каждой → пустая attachment-строка кэшировалась в `campaign.attachments` → все посты текстом (текст шёл, т.к. `wall.post` фолбэчит на user-токен).
- **Probe** `scripts/probe_wall_upload_token.py` (read-only, `getWallUploadServer` ничего не меняет) на проде: **16/16 целей сети заливаются user-токеном, 0/16 community** (все [27]).
- **Фикс:** грузим **user-токеном** админа + **по каждой целевой группе отдельно** (owner фото = эта группа; одну строку на все цели нельзя — owner-mismatch, VK дропает); кэш `campaign.attachments` стал per-group JSON-картой `{gid: 'photo..'}`. Тот же латентный баг исправлен в `ad_cabinet` (`_build_wall_attachment`/`_upload_request_photos`, 0 публикаций → не проявлялся). DM-путь (`getMessagesUploadServer`, community-токен валиден) не тронут. +новые тесты (per-target attach, parse-map, text-only/no-rebuild). Деплой без миграции (restart web+worker).
