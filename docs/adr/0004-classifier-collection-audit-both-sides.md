# ADR-0004: Классификатор видит обе стороны сбора (collection audit), а не только опубликованное

**Date:** 2026-07-07
**Status:** Accepted (решение владельца 2026-07-07: вариант B — аудит обеих сторон, shadow)
**Drives / связано:**
- [ADR-0003](0003-hitl-content-classifier.md) — HITL-классификатор (shadow-first, файл-корректировщик)
- Находка сессии 2026-07-07: источник классификатора = `bulletin_curation_runs.candidates`

## Контекст

HITL-классификатор (ADR-0003) в реальности читал `bulletin_curation_runs.candidates`.
Разбор кода ([`modules/curation/recorder.py`](../../modules/curation/recorder.py) docstring §17-19)
показал: этот журнал паркует **ровно `posts_included`** — посты, **реально попавшие в
опубликованную сводку**, то есть уже прошедшие все 9 детерминированных фильтров
[`advanced_parser._filter_post`](../../modules/vk_monitor/advanced_parser.py). Итог:

- Классификатор **слеп к пере-фильтрации** — хорошие посты, которые фильтр выбросил зря,
  нигде не сохраняются (отсев идёт в памяти, дропы только инкрементят счётчики `self.stats`).
- Видит только **пере-публикацию** — мусор, который фильтр пропустил (это и ловит оператор:
  «Продам ВАЗ», «Новости Балтасей» оказались в сводке → фильтр их пропустил).

Задумка владельца — «все собранные посты попадают на обработку нейросетью, я одобряю или
перенаправляю» — требует видеть **весь сбор**, обе стороны.

**Урок precision-16%** (ADR-0003): прошлый LLM-фильтр релевантности дал 16% точности → свёрнут.
Поэтому — **не** делаем классификатор живым гейтом сразу. Сначала shadow-аудит обеих сторон,
измеряем agree-rate, и лишь потом (отдельное решение) — enforce.

## Решение (вариант B — shadow-аудит обеих сторон)

Новый shadow-журнал **`collected_post_audit`** фиксирует каждый собранный пост с решением
фильтра — `kept` (прошёл, кандидат в публикацию) или `dropped` + причина. Классификатор читает
его вместо журнала опубликованного → видит обе стороны.

### A. Захват — на границе, без изменения логики фильтра

**Инвариант: никогда не ломать публикацию** (как `recorder.py`). Поэтому активный
`_filter_post` **не трогаем логически**:

- В двух циклах сбора ([`advanced_parser.parse_posts_from_communities`](../../modules/vk_monitor/advanced_parser.py) —
  путь районов; и `filter_posts_list` — путь областных) под гейтом накапливаем **сырой собранный
  батч**, `all_posts` = множество `kept`. После цикла — один вызов fail-safe рекордера.
- Рекордер [`modules/curation/collection_audit.py`](../../modules/curation/collection_audit.py):
  изолированная `AsyncSessionLocal()`, **never-raises** (сбой аудита не влияет на сбор/публикацию),
  **gated** (`COLLECTION_AUDIT_SHADOW_ENABLED` + allowlist регионов `COLLECTION_AUDIT_REGION_CODES`).

### B. Причина отсева — пере-вывод теми же чистыми функциями

Рекордер сам определяет причину `dropped`, прогоняя те же чистые проверки, что и `_filter_post`
(`is_advertisement`, `check_blacklist`, region-words, no-attachments) на unwrap-нутом посте
(`clear_copy_history` → тот же `lip`). Плюс: `_filter_post` остаётся нетронутым (минимальный риск).
Минус: дублирование логики (при изменении фильтров синхронить — coupling отмечен здесь и в модуле).

- **Механические отсевы** (возраст, дедуп по lip/тексту/медиа, black_id) — **пропускаем**: это
  не «пере-фильтрация», а корректный шум (иначе дедуп затопит ленту). Записываем только `kept`
  и content-дропы (`advertisement` | `blacklist_text` | `no_region_words` | `no_attachments`).

### C. Классификатор читает аудит

[`modules/classifier/service.py`](../../modules/classifier/service.py) `fetch_pending` /
`_recent_candidates` читают `collected_post_audit` (fallback на `bulletin_curation_runs`, если
аудит пуст — переходный период). Снапшот поста в ленте несёт `decision` + `drop_reason` → оператор
видит «🚫 отсеян: реклама» vs «✅ прошёл фильтр». Его несогласие на dropped («зря выбросили») =
сигнал пере-фильтрации; на kept («мусор») = сигнал пере-публикации.

### D. Идемпотентность и объём

- Ключ — `lip` (unique), first-seen wins (пост, увиденный повторно, пропускается).
- Объём вырастет (~×10 к опубликованному): весь сбор минус механические дропы. В shadow это
  материал для agree-rate; для оператора — больше ленты (осознанный trade-off владельца).

## Фазы

- **Ф1 (этот ADR):** shadow-аудит обеих сторон. Классификатор только смотрит. Enforce ВЫКЛ.
- **Ф2 (позже, отдельное решение):** при устойчиво высоком agree-rate — enforce: вердикт+одобрение
  оператора реально решают publish/delete (вариант C). Детерминированный фильтр правится по данным
  аудита (какой фильтр слишком строг/дыряв).

## Последствия

**Плюсы:** полная видимость сбора (обе стороны); безопасно (shadow, fail-open, gated); `_filter_post`
не тронут (нулевой риск для публикации); данные для калибровки детерминированного фильтра.

**Минусы:** дублирование логики причин-отсева (coupling с `_filter_post`); рост объёма ленты; ещё
одна shadow-таблица (aging по мере надобности).

**Безопасность:** рекордер never-raises + gated; аудит ничего не публикует/удаляет; тексты — снапшот
(как `content_classifications`).
