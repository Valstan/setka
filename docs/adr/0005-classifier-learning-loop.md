# ADR-0005: Петля обучения классификатора — выученные правила overlay + веб-утверждение

**Date:** 2026-07-08
**Status:** Accepted (построено и задеплоено 2026-07-08)
**Drives:** запрос владельца 2026-07-08 «замкнуть петлю: мои правки в ленте → файл обучения → нейросеть лучше понимает, но я остаюсь в петле и утверждаю».
**Расширяет:** [ADR-0003 §E/§F](0003-hitl-content-classifier.md) (файл-корректировщик + shadow-гейт).

## Контекст

ADR-0003 §E спроектировал файл-корректировщик (`config/classification_postulates.md`) как
дистиллированные обобщённые правила, пополняемые из коррекций оператора. Но petля была
**разомкнута**: рутина читала файл каждый прогон (✅), коррекции копились в
`classification_corrections` (✅), но шаг «коррекции → правила в файле» был **ручным** (правил
markdown человек коммитом). Владелец попросил замкнуть петлю, с двумя жёсткими требованиями:

1. **Человек в петле.** Нейросеть не правит правила по своему усмотрению — только предлагает;
   применяет оператор.
2. **Утверждение через веб.** Оператор утверждает правила прямо в ленте `/classifier`, а не
   через git-PR — как продолжение того же рабочего процесса, где он разбирает вердикты.

## Решение

### Слой выученных правил (БД-overlay поверх git-базы)

- **Базовый слой** — `config/classification_postulates.md` (git, hand-authored). Не меняется.
- **Выученный слой** — таблица `classification_rules` (миграция 057): `region_code` (NULL =
  глобальное), `rule_text`, `status` (`proposed|approved|rejected|retired`), `source`
  (`routine|operator`), `rationale`, `evidence` (JSONB — какие коррекции породили), `norm_key`
  (дедуп), `decided_at`, `last_effective_at` (aging).
- **Эффективные постулаты** = база + все `approved` правила (`rules.render_effective_postulates`).
  `/api/classifier/postulates` отдаёт именно их → рутина классификации автоматически видит
  утверждённые правила со следующего прогона. Пусто выученных → база байт-в-байт (prompt-cache цел).

### Петля (человек в петле дважды)

```
оператор правит вердикт в ленте  →  classification_corrections (несогласие)
        ↓  дистилляция (облачная рутина, GET /corrections + GET /postulates)
черновик правила (status=proposed)  →  POST /rule-proposals  →  панель «Правила обучения»
        ↓  оператор: ✅ утвердить / ✏️ правка+утв. / ❌ отклонить  (POST /rules/{id}/…)
approved  →  эффективные постулаты  →  рутина классификации читает след. прогоном
```

- **Слой 1 (вердикты):** оператор смотрит каждый пост, ничего не публикуется само (shadow).
- **Слой 2 (правила):** нейросеть только **предлагает** правило (из коррекций оператора),
  оператор утверждает/правит/отклоняет. В файл-overlay попадает только после «ок».

### Дистилляция без Claude API (вариант B, как классификация)

`ANTHROPIC_API_KEY` пока нет → дистилляцию делает **вторая облачная рутина** (или второй прогон):
`GET /corrections` (несогласия + снапшот поста) + `GET /postulates` (текущие правила, чтобы не
дублировать) → чеканит обобщённые черновики → `POST /rule-proposals`. Дедуп по `norm_key` против
активных (`proposed|approved`). Когда появится ключ — рутину меняем на Celery-таск (те же таблицы/
эндпоинты), как и классификацию.

### Оператор может писать правило руками

`POST /rules/add` → `status=approved`, `source=operator` — сразу в деле. Не ждать, пока рутина
догадается: очевидное правило оператор формулирует сам.

## Дивергенция от ADR-0003 §E (и почему)

ADR-0003 §E: «правила в git-файле, откат = git revert». **Живое веб-утверждение это требование
ломает** — web-процесс не может (и не должен) git-коммитить/пушить на каждый клик (нарушает
PR-only, ADR-0002; и фрагильно). Поэтому **выученный слой живёт в БД** (живая петля), а
git-версионирование сохраняется двумя способами:

1. **Базовый файл остаётся в git** (hand-authored ядро).
2. **Периодический снапшот** утверждённых правил в `classification_postulates.md` (или companion) —
   для аудита/отката (ручной MVP; beat-джоба позже). `retire` в UI = мягкий откат правила без git.

Trade-off принят осознанно: живая петля через веб важнее строгого git-версионирования выученного
слоя; аудит-снапшот закрывает потребность в истории.

## Aging (ADR-0003 §E, pool #033)

`last_effective_at` пишется, когда правило подаётся в эффективные постулаты (задел). Правило без
срабатываний N месяцев → кандидат на `retire` (ручной MVP; авто-подсветка позже). Потолок ~100
правил — похожие сливать при дистилляции.

## Последствия

**Плюсы:** петля замкнута — каждая утверждённая правка делает промпт точнее; человек в петле дважды;
работает без Claude API (рутина-мост); откат правила = `retire` в один клик.

**Минусы / trade-offs:** выученный слой вне git (митигируется снапшотом); вторая рутина жжёт токены
на дистилляцию (мало — раз в день на одном районе); дисциплина aging обязательна, иначе overlay
раздуется и деградирует (и ломает prompt-cache).

**Безопасность:** ingest-эндпоинты (`/corrections`, `/rule-proposals`) — под X-API-Key рутины;
операторские (`/rules/*`) — под сессией (не в `PUBLIC_PREFIXES`); правило применяется только после
явного утверждения оператором (человек в петле).
