## Контекст

Владелец свернул Claude-Code-рутины классификации постов (permission-friction, «до конца не доводит»)
и заказал headless-механизм. GO владельца 2026-07-05, каркас утверждён brain (5 опор), детали проектирует
setka.

**Что классифицировать:** тема поста, действие (публиковать/удалить/отложить), объединение постов
**по смыслу** (лечит склейки «спорт+похороны+церковь»). Оператор не «обучает» нейронку напрямую, но даёт
пофактовые поправки — они должны накапливаться и улучшать классификацию.

**Заземление на код (карта 2026-07-05):**
- **Claude API в проекте НЕТ** — чистый старт. Env-паттерн `_collect_prefixed_tokens` (`config/runtime.py`)
  готов под `ANTHROPIC_API_KEY` (#008).
- **Свёрнутая курация** (`BulletinCurationRun` / `bulletin_curation_runs`, gate-locked env, PoC 2026-06-07) —
  **другая гранулярность** (keep/drop постов внутри опубликованной свод­ки, не пер-пост тема+действие+merge).
  Переиспользуем **паттерн** (best-effort изолированная сессия + env-гейт + JSONB verdicts + shadow-флаг),
  **не таблицу**.
- **Посты** живут в `Post` (`database/models.py`): `text`/`attachments`/`ai_category`(тема)/`status`
  (`new→analyzed→approved/rejected/published`) + фингерпринты дедупа. Классификатор читает `Post`.
- **Темы:** 20 каноничных `POSTOPUS_BULLETIN_THEMES` (`modules/bulletin_pipeline_settings.py`) —
  novost/kultura/sport/reklama/admin/… В shadow **не констрейним** (решение владельца, см. §Решения).
- **`/requests`** = рекламный кабинет (`ad_requests`, `web/api/ad_cabinet.py`). Лента вердиктов — **новая**
  операторская поверхность рядом, не тот же роутер.

**Урок precision-16%:** LLM-фильтр релевантности setka дал 3/19 ≈ 16% на спот-чеке → свёрнут. Поэтому
shadow-first обязателен: measure-before-enforce.

## Решение (shadow-фаза Ф1)

### A. Классификатор = Claude API из Celery (не рутины)

Новый модуль `modules/classifier/`:
- **`client.py`** — тонкая обёртка над `anthropic` SDK (`client.messages.parse()` со схемой → валидный
  JSON без ручного парсинга). Ключ — `ANTHROPIC_API_KEY` из env (#008). Ядро отделено от Celery (тестируемо
  как `gateway_mcp` R18: httpx-мок, без запуска воркера).
- **Тиринг моделей** (канон brain): **`claude-haiku-4-5`** ($1/$5 за Mtok) для массы; эскалация на
  **`claude-opus-4-8`** только при низкой `уверенности` (порог ~60). Escalation-rate логируется.
- **Prompt caching** (главный рычаг стоимости): system-промпт + файл-корректировщик **байт-стабильны** →
  один `cache_control`-breakpoint кэширует их (~0.1× read); варьируется только текст поста после
  breakpoint. NB: минимум кэшируемого префикса Haiku = **4096 токенов** — пока файл правил мал, кэш
  не сработает (не ошибка, просто полная цена); окупится по мере роста правил.
- **Kill-switch** `CLASSIFIER_DISABLED` + env-гейт `CLASSIFIER_SHADOW_ENABLED` + allowlist регионов
  `CLASSIFIER_REGION_CODES` (обкатка на одном районе — решение владельца).
- Вызов дешёвый и не в latency-критичном пути → **синхронный per-post** в shadow (Batches API 50%/≤1ч —
  оптимизация на потом, когда объём вырастет; в ADR как forward-note, не строим сейчас).

### B. Схема вердикта (Pydantic → structured output)

```
ClassifierVerdict:
  theme: str            # СВОБОДНАЯ строка в shadow (не enum) — учимся, какие темы LLM рождает
  action: publish|delete|hold
  merge_with: list[int] # id постов для склейки по смыслу (пусто = самостоятельный)
  split: bool           # текущий пост — ошибочная склейка, разъединить
  confidence: int       # 0..100 (драйвер эскалации Haiku→Opus и порога enforce)
  reasoning: str        # 1 строка обоснования
```

`theme` свободной строкой (решение владельца) — на enforcing-этапе Ф2 констрейним к устаканенному словарю
(канонические 20 + добор из реально порождённых). Merge-by-meaning — тот же конвейер, отдельный тип
вердикта (лечит склейки).

### C. Shadow-таблица + лог коррекций (новая, миграция 053 → 054)

> **Правка при деплое 2026-07-05:** таблица `posts` оказалась ПУСТА — активный конвейер SARAFAN не пишет
> пер-пост `Post`-строки, а копит кандидатов внутри свод­ок (`bulletin_curation_runs.candidates` JSONB:
> `{lip, url, text, post_id, owner_id, has_media}`). Живой источник — именно он. Поэтому классификатор
> читает кандидатов свод­ок, а таблицы ключуются по **`lip`** (`"<owner_abs>_<post_id>"`, структурный
> фингерпринт), а не по `post_id`-FK на пустую `posts`; текст/url хранятся снапшотом (кандидат
> транзиентен). Миграция 054 пересоздаёт таблицы под lip (053 была пустая).

- **`content_classifications`** (пер-пост вердикт): `id`, `lip` (unique), `region_code`, `post_text`/`post_url`
  (снапшот), `source` (routine|api), `model`, `verdict` JSONB (схема B, `merge_with` — список lip),
  `confidence`, `shadow` (default true), `escalated`, `tokens_estimate`, `created_at`. Классификатор
  **только пишет** — контент не трогает в shadow.
- **`classification_corrections`** (лог несогласий, растёт из UI): `id`, `classification_id`(FK), `post_id`,
  `verdict_type` (theme|action|merge), `ai_value` JSONB, `operator_value` JSONB, `created_at`. Это сырьё
  для (1) метрики agree-rate по типам, (2) периодической дистилляции в файл-корректировщик. Родня deny-лог
  pool #054 («лог реальности → правила»).

### D. Лента вердиктов (web-UI, рядом с /requests)

Новая операторская поверхность (`web/api/classifier.py` + `web/templates/classifier.html`, меню «Система»
или вкладка рядом с рекламным кабинетом): пост → вердикт нейронки → кнопки:
- ✅ **Согласен** (один тык, пишет agree в лог без изменения вердикта);
- **Изменить тему** (свободный ввод / автодополнение по уже встреченным темам);
- **Не удалять → публиковать** / **не публиковать → удалить** (action-поправка);
- **Разъединить** (split) / **Объединить с …** (merge, выбор постов).

Каждое несогласие → строка в `classification_corrections`. Своё «состояние обработки» поверх ленты (R5),
чтобы не переклассифицировать разобранное.

### E. Файл-корректировщик («классификационные постулаты»)

`config/classification_postulates.md` (в репо, версия = git) — **дистиллированные ОБОБЩЁННЫЕ правила**,
подаётся в каждый промпт классификатора:
- **Дистилляция, не сырьё:** периодический прогон (ручной MVP / beat позже) сворачивает накопленные
  коррекции в обобщённые правила («упоминание соболезнований/прощания → тема "события", НИКОГДА не
  объединять со спортом»), не построчный лог.
- **Потолок ~100 правил / несколько КБ:** похожие сливаются; правило без срабатываний N мес — кандидат на
  удаление (зеркало aging #033). Иначе файл съест контекст и деградирует (и ломает prompt-cache).
- Откат правила = `git revert`.

### F. Shadow-гейт (measure-before-enforce)

- **Метрика:** agree-rate = agrees / (agrees + corrections) **по каждому типу вердикта отдельно**
  (theme / action / merge) — считается из `classification_corrections` + agree-логов.
- **Enforce по-типам:** авто-действие включается для типа при устойчиво высоком agree — **ориентир ≥90%
  на ≥2 недели** (калибруем по факту). Тема может стать авто раньше, чем удаление.
- **Kill/пересмотр:** agree-rate < ~60% после итераций файла правил → остановиться, не тащить в enforce
  волевым решением (возможно классы тем слишком размыты для LLM без файн-тюна).

## Оценка стоимости (порядок величины, один район)

Haiku 4.5 $1/$5 за Mtok. Пер-пост: system+постулаты ~2–4K токенов (кэш-read 0.1× после прогрева) + текст
поста ~400 токенов + вердикт-выход ~150 токенов → **≈ $0.0015 / пост**. Один район ~50–200 постов/день →
**≈ $0.1–0.3 / день ≈ $3–9 / мес**. Эскалация на Opus ~10% низко-уверенных → +копейки. Вывод: shadow на
одном районе — единицы долларов в месяц; расширение линейно по регионам.

## Фазы

- **Ф0 (этот ADR):** дизайн. Постройки нет.
- **Ф1 shadow (после ключа):** `modules/classifier/` (client+schema) + миграция 053 (2 таблицы) +
  Celery-таск (читает `Post` одного района → пишет вердикты) + web-лента + agree/correction API +
  `classification_postulates.md` (стартовый) + тесты (httpx-мок, гейт, схема). Enforce ВЫКЛ.
- **Ф1.5:** дистилляция коррекций → правила (ручной прогон), перезамер agree-rate.
- **Ф2 enforce (по-типам):** при agree ≥90%/≥2нед по типу — авто-действие этого типа; констрейн темы к
  устаканенному словарю. Merge-by-meaning в проде.
- **Этап 2 (после обкатки, НЕ сейчас):** тот же движок наполняет малмыж-сайты из ВК (вмалмыже.рф ←
  «Малмыж инфо»): классификатор отбирает + LLM верстает text↔image interleaving (рецепт Sabantuy). Канал —
  VK-шлюз #062.

## Решения владельца (2026-07-05)

1. **API-ключ — пока не провижинится:** **дизайн-фаза сейчас**, постройка Ф1 после появления
   `ANTHROPIC_API_KEY` (спенд-решение отложено). Без ключа классификатор не запускается.
2. **Темы — свободной строкой** в shadow (не констрейним 20 канон сразу): учимся, какие темы LLM рождает,
   до закрепления таксономии на enforce.
3. **Источник — один район** для обкатки (малый объём, дёшево, быстрая обратная связь по agree-rate).

## Последствия

**Плюсы:** headless 24/7 без permission-промптов; measure-before-enforce (не повторяем precision-16%);
самоулучшение через файл правил под git-версией; переиспользуемый движок для Этапа 2 (малмыж-сайты).

**Минусы / trade-offs:** новая внешняя зависимость (Anthropic API) + рекуррентный спенд (мал на одном
районе); файл правил требует дисциплины aging, иначе деградирует; таксономия тем на enforce — отдельное
решение.

**Безопасность:** `ANTHROPIC_API_KEY` — секрет только в env (#008), не в репо/чат; shadow-гейт + kill-switch;
классификатор в shadow ничего не удаляет/публикует.
