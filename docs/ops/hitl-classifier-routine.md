# HITL-классификатор: облачная рутина (этап B) — настройка

> **Что это.** Пока нет `ANTHROPIC_API_KEY` (Claude API из Celery — этап enforce), классификацию в
> shadow делает **облачная рутина** (scheduled cloud agent на claude.ai/code): по расписанию забирает
> батч постов из setka, классифицирует, возвращает вердикты. Крутится в облаке Anthropic → не зависит от
> включённого компа, ноль permission-промптов на боксе. Данные ходят через HTTP-интерфейс
> `/api/classifier` (защита — X-API-Key рутины). Дизайн — [ADR-0003](../adr/0003-hitl-content-classifier.md),
> вариант B.

## Что нужно на проде (один раз)

1. **Секрет рутины** в `/etc/setka/setka.env` (#008):
   ```
   CLASSIFIER_INGEST_KEY=<длинная-случайная-строка>
   CLASSIFIER_REGION_CODES=mi          # один район для обкатки (код региона)
   # CLASSIFIER_DISABLED=1             # аварийный kill-switch (по умолчанию выкл.)
   # CLASSIFIER_SOURCE_DAYS=3          # окно свежести /pending (сутки, дефолт 3)
   # CLASSIFIER_PENDING_MAX=40         # потолок постов в одном батче (дефолт 40)

   # Аудит сбора (ADR-0004, вариант B) — источник ОБЕИХ сторон сбора:
   COLLECTION_AUDIT_SHADOW_ENABLED=1   # включить запись собранных постов
   COLLECTION_AUDIT_REGION_CODES=mi    # тот же район, что у классификатора
   # COLLECTION_AUDIT_RETENTION_DAYS=60 # ретеншн журнала аудита (дефолт 60 сут.)
   ```
   Сгенерировать ключ: `openssl rand -base64 32`. Аудит сбора пишется в Celery-воркере
   (там идёт сбор) → после включения нужен `systemctl restart setka-celery-worker` +
   `setka` (web читает аудит). Fail-safe: сбой аудита НЕ ломает сбор/публикацию.

   **Свежесть и ретеншн.** `/pending` отдаёт только посты, собранные за последние
   `CLASSIFIER_SOURCE_DAYS` суток (дефолт **3** — «не старше 3 дней», по `collected_at`).
   Ширина окна НЕ влияет на токены: каждый пост классифицируется ровно один раз (дедуп по
   `lip` в `content_classifications` — навсегда), окно решает лишь, какие посты видны. Beat
   `prune-collected-post-audit-daily` (03:45 MSK) чистит `collected_post_audit` старше
   `COLLECTION_AUDIT_RETENTION_DAYS` (дефолт 60). Реестр вердиктов `content_classifications`
   НЕ чистится — иначе классифицированный пост воскрес бы в `/pending`.
2. **Миграция 053** (`053_hitl_classifier_shadow.sql`) применена + `restart web`.
3. Проверка снаружи (с ключом):
   ```
   curl -s -H "X-API-Key: $KEY" "https://вход.вмалмыже.рф/api/classifier/pending?limit=3"
   ```
   должен вернуть `{"count": N, "posts": [...]}`.

## HTTP-контракт (что зовёт рутина)

Базовый URL для **облачной рутины** — ASCII-хост `https://3931b3fe50ab.vps.myjino.ru`
(тот же VPS; punycode-IDN `xn--b1ae3a1a.xn--80adkdyec4j.xn--p1ai` = `вход.вмалмыже.рф` тоже
работает, но чаще спотыкается об egress-прокси облачного окружения — см. Troubleshooting ниже).
При переезде VPS myjino-хост меняется — сверяться с `docs/GATEWAY.md`. Все ingest-эндпоинты —
заголовок `X-API-Key: <CLASSIFIER_INGEST_KEY>`.

**Источник постов** — аудит сбора `collected_post_audit` (ADR-0004, вариант B): при сборе каждый
пост парков­ается с решением фильтра — `kept` (прошёл, кандидат в публикацию) или `dropped` +
причина (`advertisement`|`blacklist_text`|`no_region_words`|`no_attachments`). Так классификатор видит
**обе стороны** (пере-публикацию и пере-фильтрацию), а не только опубликованное. Ключ поста — **`lip`**
(`"<owner_abs>_<post_id>"`, напр. `156168183_14260`). Fallback на `bulletin_curation_runs.candidates`
(только опубликованное), пока аудит не наполнился. Ответ `/pending` несёт `decision` + `drop_reason`.

| Метод | Путь | Назначение |
|---|---|---|
| `GET` | `/api/classifier/pending?region=<код>&limit=<N>` | Батч постов без вердикта. `region` пуст → берётся `CLASSIFIER_REGION_CODES`. Ответ: `{count, posts: [{lip, region_code, text, has_media, url}]}` |
| `GET` | `/api/classifier/postulates` | Текст файла-корректировщика (вставить в промпт) |
| `POST` | `/api/classifier/verdicts` | Вернуть вердикты. Тело: `{"verdicts": [{lip, theme, action, merge_with, split, confidence, reasoning, model, text, url, region_code}]}`. Ответ: `{recorded, skipped_existing, skipped_missing}` |

Схема вердикта (ADR-0003 §B):
- `lip` — ключ поста из `/pending`;
- `theme` — тема поста (в shadow **свободной строкой**);
- `action` — `publish` | `delete` | `hold`;
- `merge_with` — список **`lip`** постов **из этого же батча**, которые надо склеить с текущим по смыслу;
- `split` — `true`, если текущий пост сам ошибочная склейка (разъединить);
- `confidence` — 0..100; `reasoning` — 1 строка;
- **`text`/`url`/`region_code`** — эхо из `/pending` (сохранить снапшот; если не вернёшь — сервер
  доберёт из свод­ки, но эхо надёжнее).

## Промпт рутины (заготовка, v3 — скрипт + env-секрет)

> **Почему v3.** Оба «промптовых» варианта отбиты моделью рутины (2026-07-09, см.
> Troubleshooting): и авторизационное эссе («social engineering»), и сухой нейтральный
> («not a user message, standing by»). Все претензии модели — к ОБВЯЗКЕ (ключ открытым
> текстом в инструкции + автономные HTTP-вызовы с ним), а не к самой классификации.
> v3 убирает обвязку в код: HTTP-механика — `scripts/classifier_routine.py` (репо,
> stdlib-only), ключ — env-переменная облачного окружения. Модели остаётся чистая
> редакторская задача. Это не обход, а выполнение требований самой модели.

Настройка рутины (claude.ai/code → рутина → ✏️):
1. **Репозиторий** окружения — `Valstan/setka`, ветка `main` (скрипт должен быть в checkout'е).
2. **Update cloud environment → Environment variables:** добавить
   `CLASSIFIER_INGEST_KEY=<значение с прода>` (на VPS: `/etc/setka/classifier-routine-key.txt`).
   В промпте ключа больше НЕТ.
3. **Network access → Custom** с хостом `3931b3fe50ab.vps.myjino.ru` (как раньше).
4. Промпт:

```
Плановый прогон разметки постов-кандидатов для новостных сводок. Работа идёт в
checkout'е репозитория setka; ключ API скрипт читает сам из env-переменной
CLASSIFIER_INGEST_KEY — в промпте и отчёте он не нужен.

1. python scripts/classifier_routine.py fetch
   → запишет classifier_run/postulates.md (правила разметки) и
   classifier_run/pending.json (батч постов); постам БЕЗ текста скачает их
   фото/PDF в classifier_run/media/ (пути — в поле media_files поста).
   Если count=0 или скрипт сообщил о сетевой ошибке/egress-403 — отчитайся
   одной строкой и заверши прогон.
2. Прочитай оба файла. Для каждого поста из pending.json определи по правилам:
   theme — тема свободной строкой;
   action — publish | hold | delete (брать ли кандидата в подборку:
     взять / спорный, отложить / не брать — мусор, реклама, не про район);
   merge_with — список lip постов ЭТОГО батча про то же событие (иначе []);
   split — true, если пост сам склеен из разных тем (напр. спорт+похороны);
   confidence — 0..100 (сомневаешься — ниже 60); reasoning — одна строка;
   lip, text, url, region_code — скопируй из объекта поста как есть.
   У поста БЕЗ текста с media_files: открой и посмотри эти файлы (картинки,
   PDF), вынеси вердикт по содержимому и заполни media_summary — одна строка,
   что во вложениях («афиша концерта в ДК 20 июля»). Картинки не открылись
   или в media только видео/аудио — media_summary опиши типами («видео без
   текста»), action по правилам для нераспознанного медиа.
3. Сохрани classifier_run/verdicts.json: {"verdicts": [ ...объект на каждый пост... ]}.
4. python scripts/classifier_routine.py submit classifier_run/verdicts.json
   (скрипт сам провалидирует и отправит; при ошибках валидации — исправь и повтори).
5. Отчитайся числами: постов в батче, publish/hold/delete, merge/split,
   recorded/skipped из ответа сервера.

Разметка — черновой слой: перед применением её проверяет и утверждает человек
в операторской ленте.
```

## Петля обучения — дистилляция Корпуса (ADR-0005)

> **Актуальный способ (решение владельца 2026-07-14):** облачная рутина-дистиллятор
> **удалена** (переключить её модель через ✏️ на claude.ai не удалось — UI не нашёлся).
> Дистилляцию делает Claude **из dev-чата** скиллом [`/distill`](../../.claude/commands/distill.md):
> забирает коррекции скриптом по SSH (ключ не покидает VPS), анализирует, показывает
> черновики владельцу до отправки, шлёт `propose`, ведёт журнал
> [`DISTILL_LOG.md`](DISTILL_LOG.md). `/start` напоминает, если прошло > 7 дней.
> **Корпус** = накапливаемые посты+вердикты+правки (`content_classifications`, вечно).
> Разделы ниже про настройку второй рутины — историческая справка (пригодятся, если
> когда-нибудь вернём автоматическую рутину).

Замыкает петлю: коррекции оператора в ленте → дистилляция чеканит **черновики** обобщённых правил →
оператор утверждает в вебе → утверждённые правила подмешиваются в `/postulates` (их читает рутина
классификации). Нейросеть правила сама не применяет — только предлагает; человек в петле.

**Отдельная облачная рутина** (или второй прогон), расписание реже классификации (напр. раз в
сутки — коррекции копятся медленно). Те же хост/ключ. Эндпоинты (все под `X-API-Key`):

| Метод | Путь | Назначение |
|---|---|---|
| `GET` | `/api/classifier/corrections?limit=100&days=30` | Несогласия оператора + снапшот поста (`{lip, verdict_type, ai_value, operator_value, post_text, post_url, region_code}`) |
| `GET` | `/api/classifier/postulates` | Текущие эффективные правила (база + утверждённые) — чтобы не дублировать |
| `POST` | `/api/classifier/rule-proposals` | Черновики правил: `{"proposals":[{rule_text, rationale, evidence:[…], region_code?, model}]}`. Дедуп по смыслу против активных. Ответ `{recorded, skipped_existing, skipped_invalid}` |

### Зачем она и как работает (объяснение владельцу)

Классификатор учится на ваших правках. Когда вы в ленте `/classifier` не соглашаетесь с ИИ
(меняете тему/действие), несогласие копится в лог коррекций. Рутина-дистиллятор раз в сутки
перечитывает эти коррекции и **предлагает** обобщённые правила («частные продажи → hold», «посты
про соседний район → delete») — черновиками в панель «Правила обучения» той же ленты. Вы их
утверждаете/правите/отклоняете; утверждённое правило со следующего прогона читает рутина
классификации — и перестаёт повторять исправленную ошибку. Без дистиллятора петля тоже живёт
(правило можно написать руками в той же панели), рутина лишь автоматизирует «перечитать правки
и обобщить». Запускать её имеет смысл, когда в ленте накопились разобранные вердикты с правками
(коррекций ≥ ~10) — на пустом логе она честно завершится «нечего дистиллировать».

### Настройка второй рутины (шаги владельца)

Всё как у первой рутины (та же схема v3: механика в скрипте, ключ в env):
1. claude.ai/code → **Routines → + New routine**. Имя, напр. «Дистиллятор правил Сарафана».
   **Модель — Opus 4.8, reasoning effort Low** (решение владельца 2026-07-13: дистилляция —
   обобщение паттернов из правок, тут важнее качество формулировки правила, чем скорость;
   Haiku чеканит правила площе. Low-усилие держит стоимость разумной — прогон раз в сутки
   на десятках коррекций). У существующей рутины модель меняется через ✏️ на её странице.
2. Расписание — **реже классификации**, коррекции копятся медленно: раз в сутки, напр.
   `50 21 * * *` (21:50 MSK, после вечернего разбора ленты).
3. Окружение: репозиторий **`Valstan/setka`**, ветка `main`; **Environment variables** →
   `CLASSIFIER_INGEST_KEY=<значение с прода>` (то же, что у первой; на VPS —
   `/etc/setka/classifier-routine-key.txt`); **Network access → Custom** с хостом
   `3931b3fe50ab.vps.myjino.ru`. Лишние коннекторы (Gamma и пр.) не подключать.
4. Промпт — заготовка ниже.

### Промпт рутины-дистиллятора (заготовка, v3 — скрипт + env-секрет)

```
Плановая дистилляция правок оператора в черновики правил классификатора.
Работа идёт в checkout'е репозитория setka; ключ API скрипт читает сам из
env-переменной CLASSIFIER_INGEST_KEY — в промпте и отчёте он не нужен.

1. python scripts/classifier_routine.py corrections
   → запишет distill_run/postulates.md (действующие правила разметки) и
   distill_run/corrections.json (несогласия оператора: verdict_type,
   ai_value → operator_value + текст поста). Если count=0 или скрипт сообщил
   о сетевой ошибке/egress-403 — отчитайся одной строкой и заверши прогон.
2. Прочитай оба файла. Найди в коррекциях ПОВТОРЯЮЩИЕСЯ паттерны (не единичные
   случаи): что оператор систематически поправляет и как. По каждому паттерну
   сформулируй одно ОБОБЩЁННОЕ правило в стиле действующих постулатов
   («если …, то action/theme …»). Не плоди правило под один пост; не дублируй
   правила, уже присутствующие в postulates.md.
3. Сохрани distill_run/proposals.json:
   {"proposals": [{"rule_text": "обобщённое правило одной-двумя фразами",
                   "rationale": "какой паттерн правок его породил",
                   "evidence": [{"lip": "…", "verdict_type": "action",
                                 "ai_value": "publish", "operator_value": "delete"}],
                   "region_code": null, "model": "<твоя модель>"}]}.
   Предлагай мало и метко: 2–5 сильных правил за прогон лучше, чем 20 слабых —
   каждое оператор читает и утверждает вручную. Нет устойчивых паттернов —
   не предлагай ничего и отчитайся об этом.
4. python scripts/classifier_routine.py propose distill_run/proposals.json
   (скрипт сам провалидирует и отправит; при ошибках валидации — исправь и повтори).
5. Отчитайся числами: коррекций прочитано, черновиков предложено,
   recorded/skipped из ответа сервера.

Правила — черновой слой: ты только предлагаешь; применяет их человек после
утверждения в операторской ленте.
```

Оператор видит черновики в ленте `/classifier` → панель **«Правила обучения»** → ✅ утвердить /
✏️ правка+утвердить / ❌ отклонить. Утверждённое — сразу в `/postulates` следующего прогона
классификации. Может и сам написать правило (поле «написать правило вручную» → сразу в деле).

### Снапшот выученных правил и aging (хвосты Б/В ADR-0005)

**Снапшот (аудит/откат выученного слоя).** Beat `snapshot-learned-rules-daily` (03:50 MSK)
сериализует approved-правила в файл на проде — дефолт `logs/classifier_learned_rules_snapshot.md`
(env `CLASSIFIER_RULES_SNAPSHOT_PATH`; путь должен оставаться **untracked** — tracked-файл,
перезаписанный beat'ом, сломал бы `git pull` грязным деревом). Контент детерминированный (без
timestamp генерации): дифф = реальное изменение слоя. **Захват в git-историю** — шаг dev-сессии
(с прода коммитить нельзя, PR-only ADR-0002): при касании нитки классификатора

```
ssh setka "cat /home/valstan/SETKA/logs/classifier_learned_rules_snapshot.md" \
  > config/classification_postulates_learned.snapshot.md   # + обычный PR
```

**Aging (#033).** При каждой подаче в эффективные постулаты approved-правилам штампуется
`last_effective_at`. Правило без штампа дольше `CLASSIFIER_RULE_STALE_DAYS` (дефолт 90;
свежеутверждённое меряется по `decided_at`) в панели «Действующие правила» получает бейдж
**«⏱ протухает»** — кандидат на 🗑 retire (мягкий вывод без git). Авто-retire нет — решает оператор.

## Что видит оператор

Лента вердиктов — `/classifier` (меню «Система» → «Классификатор»). Пост + вердикт нейронки + кнопки
✅ Согласен / Изменить тему / → публиковать|удалить|отложить / Разъединить. Несогласия копятся в лог,
наверху — **agree-rate по типам** (тема/действие/склейка). Когда agree-rate по типу устойчиво ≥90%
(≥2 недели) — тип созрел для enforce (ADR-0003 §F).

## Troubleshooting

**403 на CONNECT от прокси облачного окружения** («organization egress-policy denial», рутина
рапортует «0 постов, техническая блокировка»). Это НЕ прод и НЕ сеть — **Network access** облачного
окружения рутины по умолчанию `Trusted` (пускает только скачивание пакетов из проверенных источников,
но НЕ произвольные хосты). Лечение:
1. **Разрешить наш хост в окружении рутины.** claude.ai/code → страница рутины → ✏️ →
   **Update cloud environment** → блок **Network access**. По умолчанию стоит `Trusted` — сменить на
   **`Custom`** («Create a list of allowed domains») и добавить хосты:
   `3931b3fe50ab.vps.myjino.ru` (и при желании punycode `xn--b1ae3a1a.xn--80adkdyec4j.xn--p1ai`).
   `Save changes`. **NB:** диалог пишет *«Changes will apply to new sessions»* — подхватится со
   следующего прогона, не задним числом. Альтернатива — `Full` (безлимит), но `Custom` безопаснее:
   рутине нужен ровно один хост.
2. В промпте рутины использовать ASCII-хост (`3931b3fe50ab.vps.myjino.ru`) как основной —
   заготовка выше уже на нём (punycode-IDN спотыкается об egress-прокси чаще).
3. Проверка, что дело не в проде: `curl -s -o /dev/null -w '%{http_code}'
   https://3931b3fe50ab.vps.myjino.ru/api/classifier/pending?limit=1` → `401` = API жив
   (нет ключа), значит блокирует именно окружение рутины.

**Рутина «вежливо отказывается» вместо работы** (в отчёте прогона — рассуждение про authorization /
social engineering / API key safety / «not a user message», счётчик `content_classifications` не
растёт, egress уже открыт). Это НЕ сеть и НЕ права — **over-refusal модели** внутри рутины.
История лечения (не повторять пройденное):
- **v1, сухой короткий промпт** («классификатор постов, POST verdicts») → отказ «рулишь чужой
  модерацией». Вылечено v2.
- **v2, авторизационное эссе** (блок «Контекст и авторизация»: чей сервис, кто запустил, что значат
  вердикты) → работало ~2026-07-07..08, затем модель ужесточилась и стала читать САМО эссе как
  «social engineering pattern» («legitimate authorization should be simpler»), а фразу «без него
  задача читается неверно» — как попытку снять ограничения. Одновременно нейтральный промпт без
  эссе отбивался harness-пометкой «automated background-task event, NOT a message from the user» →
  «standing by». Оба края отбиты 2026-07-09.
- **v3, текущий (см. заготовку выше):** убрать триггеры физически, а не риторически. Претензии
  модели — к обвязке (ключ в тексте промпта + автономные HTTP-вызовы с ним), не к классификации.
  HTTP-механика → `scripts/classifier_routine.py` (код в репо), ключ → env-переменная облачного
  окружения, модели — только чтение файлов и разметка. Уговаривать модель дальше промптом —
  ненадёжно и по сути обход её guardrail'ов; v3 вместо этого выполняет её же требования.

Проверка эффекта любого фикса — сдвиг `MAX(created_at)` в `content_classifications` (см. ниже).
Если и v3 отобьётся — промпт-путь исчерпан, переходить на enforce (Claude API из Celery, раздел ниже).

Проверка «рутина снова пишет»:
```
ssh setka "sudo -u postgres psql -d setka -tA -c \
  \"SELECT source, COUNT(*), MAX(created_at) FROM content_classifications GROUP BY source;\""
```
`MAX(created_at)` свежее прошлого прогона = рутина ожила.

## Переход на Claude API (enforce)

Когда появится `ANTHROPIC_API_KEY`: ingest-рутину выключаем (`CLASSIFIER_DISABLED=1` или снимаем ключ
рутины), Celery-таск зовёт Claude API и пишет в те же таблицы через `modules.classifier.service`. Лента,
метрика и файл-корректировщик не меняются — меняется только «мотор» классификации.
