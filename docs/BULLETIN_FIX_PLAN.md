# План исправлений парсинга-фильтрации-постинга

## Проблемы (из тестового прогона)

### 1. Обрезание постов на полуслове
**Симптом:** Сводка обрывается `"лет.В детс..."` — пост обрезан по лимиту 4096 символов на полуслове.
**Причина:** `BulletinBuilder._available_length()` резервирует место, но `truncate_text()` режет по символам без учёта границ предложений.
**Решение:** НЕ добавлять пост если он целиком не влезает. Если post + attribution + spacing > available_length — пропустить пост, оставить на следующую итерацию.

### 2. Публикация не тем токеном
**Симптом:** Пост публиковался токеном `Vita` (попал в предложенные, не опубликовался).
**Причина:** Тестовый скрипт берёт первый токен из `VK_TOKENS` без проверки роли.
**Решение:** 
- Ввести понятие ролей токенов: `PUBLISH_TOKENS` и `PARSE_TOKENS`
- Публикация ТОЛЬКО через `VALSTAN` токен
- Парсинг может использовать любой токен
- Добавить в `config/runtime.py` явное разделение
- В `VKPublisher` принимать только publish-токен

### 3. Форматирование сводки — "сплошное мясо"
**Симптом:** Все посты слиты в один блок без визуального разделения.
**Причина:** Разделитель — только пустая строка. Нет эмодзи-маркеров начала поста, нет чёткого разделения.
**Решение (формат из old_postopus):**
```
{HEADER}

✍ {post_text_1}

@https://vk.com/wall-... (source_name)

✍ {post_text_2}

@https://vk.com/wall-... (source_name)

#хештег_темы #хештег_региона
```
- Каждый пост начинается с `✍ ` (эмодзи-маркер)
- После текста поста — пустая строка
- Строка источника `@url (имя)` 
- Пустая строка-разделитель
- В конце — хештеги темы и региона

### 4. Перемешивание негативных и позитивных новостей
**Симптом:** Новости о погибших на СВО идут вперемешку с позитивными новостями.
**Причина:** Нет разделения по тональности при формировании сводки.
**Решение (многоуровневое):**

#### 4a. Базовый уровень — keyword-based фильтрация (уже есть!)
- Использовать существующий `SentimentAnalyzer` (`modules/ai_analyzer/sentiment_analyzer.py`)
- Добавить `NEGATIVE_MARKERS` для новостей о смерти: `погиб`, `умер`, `скончался`, `смерть`, `трагедия`, `прощание`
- Перед построением сводки:
  - Анализировать sentiment каждого поста
  - Группировать: `mourning` (скорбь), `positive`, `neutral`, `negative_other`
  - Формировать **отдельные сводки** для mourning новостей

#### 4b. Создание `MourningBulletinBuilder`
- Отдельный билдер для траурных новостей
- Формат:
```
🕯 Скорбим

https://vk.com/wall-... (source)
Текст новости о погибшем...
@https://vk.com/wall-... (source)

Выражаем соболезнования родным и близким.
```
- Публикуется **отдельным постом**, НЕ вперемешку с обычными новостями

#### 4c. Обычная сводка — БЕЗ mourning новостей
- Исключить посты с `sentiment=mourning` из обычной сводки
- Обычная сводка содержит только positive/neutral/negative-other

#### 4d. (Опционально) Qwen API для более точного анализа
- Подключить Qwen API через DashScope для анализа тональности
- Более точное определение sentiment чем keyword-based
- Настроить отдельный модуль `modules/ai_analyzer/qwen_sentiment.py`
- Fallback на keyword-based если API недоступен

## Этапы выполнения

### Этап 1: Разделение токенов (PUBLISH vs PARSE)
**Файлы:** `config/runtime.py`, `test_parse_run.py`, `tasks/parsing_scheduler_tasks.py`
1. В `config/runtime.py`:
   - `VK_PUBLISH_TOKENS` — только `VALSTAN`
   - `VK_PARSE_TOKENS` — все токены (включая VALSTAN для парсинга)
2. В `test_parse_run.py` — использовать VALSTAN для публикации
3. В `VKPublisher` — валидация что токен из publish-списка

### Этап 2: Форматирование сводки (эмодзи-разделители)
**Файлы:** `modules/publisher/bulletin_builder.py`, `utils/post_utils.py`
1. Добавить `✍ ` перед каждым постом
2. Порядок: `✍ текст` → пустая строка → `@url (source)` → пустая строка
3. Убрать `truncate_text` — НЕ обрезать посты, пропускать если не влезают
4. `_available_length()` — проверять полный пост + overhead

### Этап 3: Sentiment-анализ и разделение сводок
**Файлы:** `modules/ai_analyzer/sentiment_analyzer.py`, `modules/publisher/bulletin_builder.py`
1. Расширить `NEGATIVE_WORDS` и `SADNESS_MARKERS` для новостей о смерти
2. Добавить новый label `mourning` (скорбь) в `SentimentAnalyzer`
3. Создать `MourningBulletinBuilder` для отдельных траурных постов
4. Добавить `BulletinSplitter` — разделяет посты по sentiment перед билдингом

### Этап 4: (Опционально) Qwen API интеграция
**Файлы:** новый `modules/ai_analyzer/qwen_sentiment.py`
1. Настроить DashScope API клиент
2. Создать `QwenSentimentAnalyzer` с fallback на keyword-based
3. Добавить env vars: `DASHSCOPE_API_KEY`
4. Тестовый прогон с Qwen API

### Этап 5: Тестовый прогон и проверка
1. Запустить полный прогон на тестовом регионе
2. Проверить:
   - Посты не обрезаются
   - Сводка отформатирован с эмодзи
   - Публикация через VALSTAN токен
   - Mourning новости в отдельном посте
3. Проверить результат в VK
