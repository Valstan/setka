# /close_session — финализация сессии SETKA

Задача: зафиксировать **state of mind** (что делаем сейчас, что следующее, какие подходы пробовали и отвергли), чтобы следующая сессия (через `/start` или с другого компа после `git pull`) мгновенно подхватила работу.

**Что НЕ делает:** не релиз, не деплой, не build, не прод-операции, не запуск тестов. Для деплоя — [`/reliz`](reliz.md). Для коммита накопленных рабочих правок без handoff — [`/finish`](finish.md). Эти три команды дополняют друг друга:

| Команда | Что фиксирует |
|---|---|
| [`/finish`](finish.md) | Рабочие правки кода/доков — DEV_HISTORY, PENDING_FOLLOWUPS, опц. commit без push |
| [`/close_session`](close_session.md) | State of mind — SESSION_HANDOFF (текущая нитка, следующий шаг), commit + push отдельным коммитом |
| [`/reliz`](reliz.md) | PR → merge → SSH `git pull` → systemctl restart → health-check |

Обычный flow конца дня: сначала `/finish` (зафиксировать рабочие правки), потом `/close_session` (зафиксировать handoff).

## Когда вызывать

- Перед паузой в работе (конец рабочего дня, переход на другой проект).
- После значимого milestone (PR merged, нитка частично закрыта).
- Перед сменой компа (закрыть на одном — открыть на другом через `git pull`).
- Перед длительным перерывом (отпуск, переключение приоритетов).
- **НЕ вызывать** — после одиночного hotfix без хвостов, после полностью завершённой задачи без новой нитки (handoff обновлять не нужно), во время активного `/reliz` (сначала доделай релиз).

## Шаг 1. Контекст

В одном параллельном блоке (read-only):

```bash
git log --oneline -10
git status --short --branch
git rev-parse --abbrev-ref HEAD
git log -1 --format='%H %s' -- docs/SESSION_HANDOFF.md
```

Параллельно:
- `Read` текущий [`docs/SESSION_HANDOFF.md`](../../docs/SESSION_HANDOFF.md) (если есть) — что было заявлено в начале сессии, чтобы сравнить факт vs план.
- `Read` [`docs/PENDING_FOLLOWUPS.md`](../../docs/PENDING_FOLLOWUPS.md) (первые 60 строк) — что нужно синхронизировать.
- Через `TaskList` (если использовался) — текущее состояние tasks этой сессии.

Если рабочее дерево чистое, последний коммит handoff'а — сегодня, и в TaskList нет открытых задач — мягко сказать «handoff уже свежий, ничего обновлять не нужно» и завершиться без коммита.

## Шаг 2. Уточнить у пользователя (через `AskUserQuestion` — один блок, max 4 вопроса)

Если из коммитов сессии и TaskList всё ясно — **не спрашивай**, сразу пиши драфт handoff'а и попроси подтверждения через короткое сообщение. Иначе:

1. **Какая нитка активна?** (если из контекста несколько вариантов) — 1-3 предложения. Если из аргумента команды (`$ARGUMENTS`) уже понятно — пропустить.
2. **Status:** `ACTIVE` (есть нитка) или `IDLE` (всё закрыто, нитки нет)?
3. **Следующий шаг?** Конкретно: первое действие следующей сессии с file paths и командами, чтобы можно было выполнять без додумывания.
4. **Failed approaches?** Только при `Status: ACTIVE` — были ли в этой сессии подходы, которые пробовали и отвергли (PoC не взлетел, обсудили и отбросили)? Если да — кратко перечислить с причиной отказа.

При `Status: IDLE` — перенеси failed approaches из старого SESSION_HANDOFF в `DEV_HISTORY.md` соответствующего блока (раздел «Failed approaches» или «Уроки»), потом перезаписывай handoff пустыми секциями.

## Шаг 3. Записать `docs/SESSION_HANDOFF.md`

**Перезаписать целиком** через `Write` по шаблону. История уже есть в `git log -- docs/SESSION_HANDOFF.md` и в `docs/DEV_HISTORY.md` — не аккумулируй здесь.

```markdown
# Session Handoff

> Sticky-note для непрерывности между сессиями разработки SETKA. Перезаписывается через [`/close_session`](../.claude/commands/close_session.md) — историю смотри через `git log -- docs/SESSION_HANDOFF.md` и [`DEV_HISTORY.md`](DEV_HISTORY.md).

**Status:** ACTIVE | IDLE
**Updated:** YYYY-MM-DD
**Branch:** <git current branch>
**Last release in prod:** <последний коммит на проде или ->

---

## Текущая нитка

<1-3 предложения: что делаем, в какой стадии, почему это длинная задача.>
<если Status: IDLE — пиши «_Нет — последняя задача закрыта, открытая стартовая позиция._»>

## Следующий шаг

<конкретно: первое действие новой сессии. С file paths и командами.>
<если Status: IDLE — перечисли 2-3 кандидатные стартовые точки из PENDING_FOLLOWUPS.>

## Контекст

- **План:** <путь к docs/plans/X.md если есть, иначе «нет активного плана»>
- **Связанные коммиты сессии:** <hashes + одной строкой о каждом>
- **Прод:** <статус: какие сервисы active, последний коммит на проде, есть ли расхождение с main>
- **Открытых PR:** <link или «нет»>

## Failed approaches (этой нитки)

<если в активной нитке пробовали подходы и отвергли — фиксировать здесь, чтобы будущая сессия не повторила. Формат:>
- **<подход X>** — попробовали в коммите [<hash>](...). Не сработало: <причина>. **Не повторять** без новой инфы.
- **<подход Y>** — отклонён на этапе обсуждения по причине <...>.

<если не было отвергнутых — оставить «_Не было — все попробованные подходы сработали или ещё в работе._»>
<если Status: IDLE — всю секцию можно убрать.>

## Открытые вопросы для пользователя

<список или «_Нет._»>

## Не забыть (low-priority)

<точечно из PENDING_FOLLOWUPS или из памяти сессии — то, что не доехало до отдельного техдолга>

---

> Если читаешь это в начале новой сессии — обнови через `/close_session` в конце. **Не аккумулируй history тут** — она в `git log` + `DEV_HISTORY.md`.
```

**Где что фиксируется (важно, чтобы не путать с другими файлами):**

| Куда | Что |
|---|---|

| [`docs/DEV_HISTORY.md`](../../docs/DEV_HISTORY.md) | Хронология сессий — что сделано, какие хвосты ушли в PENDING. Растёт сверху. |
| [`docs/PENDING_FOLLOWUPS.md`](../../docs/PENDING_FOLLOWUPS.md) | Открытые задачи и техдолги с приоритетами 🔴⏳🟡🟢. |

| `../brain_matrica/adr/` | Архитектурные решения (с отвергнутыми альтернативами в секции «Alternatives considered»). |

## Шаг 4. Синхронизировать `PENDING_FOLLOWUPS.md` (если нужно)

Если в сессии:
- **Закрыли** пункты — вычеркнуть (`~~...~~ закрыто YYYY-MM-DD, см. DEV_HISTORY`) или перенести подробности в `DEV_HISTORY.md`.
- **Появились новые** техдолги / идеи — добавить с приоритетом 🔴⏳🟡🟢.
- **Изменился приоритет** — переставить (например, 🟡 → 🔴 если стало блокером).

Не дублировать содержимое handoff'а — handoff **ссылается** на конкретные пункты PENDING, не повторяет.

## Шаг 5. Commit + push

**Один отдельный коммит** про handoff:

```bash
git add docs/SESSION_HANDOFF.md docs/PENDING_FOLLOWUPS.md
git commit -m "$(cat <<'EOF'
chore(session): handoff — <одна строка о нитке или «закрытие сессии IDLE»>

<2-3 строки развёрнутого описания: что сделано в сессии, какая нитка остаётся активной (или почему IDLE), что следующее.>

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
EOF
)"
```

**Direct push в `main` запрещён** ([ADR-0002](../../../brain_matrica/adr/0002-pr-only-flow-no-direct-push.md)). Алгоритм для веток:

- **Если на main** — handoff коммит маленький и тематический, всё равно через PR. Создать feature-ветку `chore/handoff-YYYY-MM-DD` (или близкий slug) → `git push -u origin <branch>` → `gh pr create` → ждать ревью → merge через `gh pr merge --squash --delete-branch`. См. порядок ниже.
- **Если на feature-ветке текущей нитки** — добавить handoff коммит к этой же ветке (не разводить два PR). `git push` — продолжение существующей.

**НЕ коммить вместе с другими изменениями** — handoff коммит обычно живёт в feature-ветке как «последний штрих перед `/reliz`» или в отдельной chore-ветке.

PR-шаблон для отдельного handoff-PR:

```bash
gh pr create --title "chore(session): handoff — <тема>" --body "$(cat <<'EOF'
## Summary

Handoff обновлён через `/close_session`:
- **Status:** ACTIVE | IDLE
- **Текущая нитка:** ...
- **Следующий шаг:** ...

## Test plan

- [x] Только doc-changes; pytest не требуется
- [x] pre-commit run --all-files Passed

🤖 Generated with [Claude Code](https://claude.com/claude-code)
EOF
)"
```

## Шаг 6. Финальный отчёт

Структура (5-8 строк, на русском):

1. **Сделано в сессии:** 1-3 строки из git log.
2. **Handoff обновлён:** показать ключевые поля (Status, Текущая нитка, Следующий шаг).
3. **Что в `PENDING_FOLLOWUPS` изменилось** (если изменилось): закрытые / новые / переприоритизированные.
4. **Коммит:** хеш + сообщение. PR URL (если открыт).
5. **Готово к закрытию терминала.**

Финальная строка — «До следующей сессии. `/start` сам подсветит нитку из handoff'а.»

## Что НЕ делать в `/close_session`

- **Не push'ить на прод** (`ssh setka`), не перезапускать сервисы, не применять миграции — это зона `/reliz`.
- **Не делать `git rebase` / `git reset --hard` / `git push --force`** — handoff коммит обычный.
- **Не direct push в main** ([ADR-0002](../../../brain_matrica/adr/0002-pr-only-flow-no-direct-push.md)) — даже handoff идёт через PR.
- **Не дублировать содержимое `PENDING_FOLLOWUPS` в handoff** — handoff ссылается на пункты, не повторяет.
- **Не вести historic log в handoff'е** — он перезаписывается. История уже в `git log -- docs/SESSION_HANDOFF.md` и в `docs/DEV_HISTORY.md`.
- **Не запускать тесты или build** — это инструмент фиксации состояния, не финализации работы.

---

> При значимых правках кода/доков в сессии — отдельный коммит через `/finish` или `/reliz` **до** `/close_session`, чтобы handoff коммит был чистым «sticky-note», а не сборной солянкой.
