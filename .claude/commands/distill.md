# /distill — дистилляция Корпуса в правила классификатора (из чата)

Цель: разобрать **Корпус** (накопленные вердикты нейросети + правки оператора в
`content_classifications`) и предложить обобщённые правила для файла-корректировщика.
Заменяет удалённую облачную рутину «Дистиллятор правил Сарафана» (решение владельца
2026-07-14): дистилляцию делает Claude прямо в dev-сессии, по запросу владельца
(обычно раз в неделю; напоминание — шаг 3.2 в `/start`).

Петля (ADR-0005) не меняется: черновики правил → панель «Правила обучения» в
`/classifier` → оператор утверждает/правит/отклоняет → approved попадает в
эффективные `/postulates` следующего прогона рутины-классификатора. ИИ правила
сам не применяет — человек в петле.

## Шаг 1. Забрать Корпус с прода (read-only)

Скрипт гоняется **на VPS** (ключ не покидает прод; в локальном env его нет).
NB: файл ключа хранит строку `CLASSIFIER_INGEST_KEY=<значение>` (KEY=VALUE, не голый
ключ) — извлекать через `cut`, иначе 401 (проверено live 2026-07-14):

```bash
ssh setka 'cd /home/valstan/SETKA && CLASSIFIER_INGEST_KEY=$(sudo -n grep -m1 "^CLASSIFIER_INGEST_KEY=" /etc/setka/classifier-routine-key.txt | cut -d= -f2- | tr -d "[:space:]") python3 scripts/classifier_routine.py corrections --limit 200 --days 30 --out /tmp/distill_run'
ssh setka "cat /tmp/distill_run/corrections.json"
ssh setka "cat /tmp/distill_run/postulates.md"
```

Если `count=0` — честно доложить «коррекций нет, дистиллировать нечего» и завершить
(журнал всё равно обновить, Шаг 5). Порог осмысленности: коррекций ≥ ~10.

## Шаг 2. Анализ (в чате, это и есть работа)

Прочитать корпус коррекций + действующие постулаты. Найти **повторяющиеся паттерны**
(не единичные случаи): что оператор систематически поправляет и как. По каждому
паттерну — одно ОБОБЩЁННОЕ правило в стиле действующих («если …, то action/theme …»).

**Гигиена файла правил (требование владельца — файл не должен разрастаться):**
- не дублировать правила, уже присутствующие в postulates.md;
- не противоречить существующим; если новый паттерн **противоречит старому правилу**
  — предложить не новое правило, а ПРАВКУ/retire старого (оператору в панели:
  «правило N устарело, паттерн сменился на …»);
- заодно просмотреть действующие выученные правила с бейджем «⏱ протухает»
  (без подачи дольше `CLASSIFIER_RULE_STALE_DAYS`) и включить в отчёт кандидатов на 🗑;
- предлагать мало и метко: 2–5 сильных правил за прогон лучше 20 слабых. Нет
  устойчивых паттернов — не предлагать ничего.

## Шаг 3. Показать владельцу ДО отправки

Вывести предлагаемые правила в чат (rule_text + rationale + сколько правок за каждым).
Дождаться «ок» (можно скорректировать формулировки прямо тут). Это дешёвле, чем
гонять черновики через панель туда-сюда.

## Шаг 4. Отправить черновики

Записать `proposals.json` локально (scratchpad), закинуть на VPS и отправить:

```bash
scp <scratchpad>/proposals.json setka:/tmp/distill_run/proposals.json
ssh setka 'cd /home/valstan/SETKA && CLASSIFIER_INGEST_KEY=$(sudo -n grep -m1 "^CLASSIFIER_INGEST_KEY=" /etc/setka/classifier-routine-key.txt | cut -d= -f2- | tr -d "[:space:]") python3 scripts/classifier_routine.py propose /tmp/distill_run/proposals.json'
```

Формат: `{"proposals": [{"rule_text", "rationale", "evidence": [{"lip", "verdict_type",
"ai_value", "operator_value"}], "region_code": null, "model": "<модель сессии>"}]}`.
Ответ сервера `{recorded, skipped_existing, skipped_invalid}` — в отчёт.

После отправки напомнить владельцу: утвердить черновики в `/classifier` →
панель «Правила обучения».

## Шаг 5. Журнал (обязателен, даже при count=0)

Добавить строку **сверху** таблицы [`docs/ops/DISTILL_LOG.md`](../../docs/ops/DISTILL_LOG.md):
дата, коррекций прочитано, черновиков предложено, recorded, заметка. Закоммитить
вместе с тематической работой или отдельным doc-PR (`/close_session` подхватит).
По этому файлу `/start` считает «дней с последней дистилляции» для напоминания.

## Шаг 6. Отчёт

Числа: коррекций прочитано / паттернов найдено / черновиков предложено /
recorded/skipped / кандидатов на retire. Плюс agree-rate из ленты, если под рукой.
